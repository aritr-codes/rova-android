## Capture / Mode

- [ ] **PR-δ FrontBack PiP / DualSight (2-sensor concurrent capture)** — ⛔ **DEFERRED / BLOCKED** (was P2)
  Build `CaptureTopology.FrontBack`: two concurrent camera bindings (front + back) composed into a single-file picture-in-picture clip via CameraX `CompositionSettings`, hardware-capability-gated. **BLOCKED pending a device with verified concurrent front+back support** (Pixel 7+ / Samsung S21+ Director's View class). A hardware probe (2026-06-14) proved the available device (Galaxy A-class, single-ISP) cannot do it at the silicon level: `FEATURE_CAMERA_CONCURRENT=false`, `getConcurrentCameraIds()` empty, `availableConcurrentCameraInfos` empty, `bindToLifecycle` threw `UnsupportedOperationException`; raw Camera2 confirmed. A follow-on analysis found **no non-concurrent path to simultaneous front+back *video*** (rapid-switch = <3fps; sequential/BeReal = photos = a different feature; LOGICAL_MULTI_CAMERA is same-direction-only; no Camera2 bypass). Owner direction: defer, do **not** build a photo fallback. `FrontBack` stays an enum value + UI refs fenced by `checkFrontBackCapabilityGated`; the gate's allowlist extension is **pending** until the feature is unblocked. Capability helper `service/dualsight/ConcurrentCameraCapability.kt` (+ 6 JVM tests) preserved for future use. **Full investigation — design spec, probe evidence + log, δ0 plan, deferred feature plan (resume preconditions), and the preserved keeper — is archived in PR #115 (branch `feat/pr-delta-dualsight`), not on master.** *(Source: ADR-0029 §5 implementation-status note 2026-06-14; PR #115 archival record.)*

- [x] **Recording start cue (beep) bleeds into the recorded audio track** — ✅ **FIXED 2026-06-18 (PR #121, branch `fix/beep-cue-bleed`, commit `9848fab`; device-smoke PASS single + DualShot).**

  On a genuinely hot device (real thermal climbing, AP ~45°C) a DualShot session can produce **asymmetric per-side output**: portrait ends with one fewer clip than landscape, and landscape can show a ~20s mid-clip freeze (a large PTS gap / near-empty segment window). **Root cause: dual-HW-encode contention (Limiter-2)** — the two concurrent H.264 encoders on this Exynos already drop ~10% of mailbox frames under contention (per `docs/superpowers/specs/2026-06-29-dualshot-fps-cadence-findings.md`); heat amplifies it, and under enough pressure one side's segment can be starved to too-few/zero frames → `MediaFileValidator` drops it, desyncing the P/L clip count. **Device evidence (2026-07-01, RZCYA1VBQ2H):** reproduced only at **full-rate + hot** (the ADR-0035 revert run); a **cool** full-rate control run was clean and symmetric (P 20.9fps/5seg, L 21.6fps/5seg, only normal ~1.4s seams). So it is **thermal/contention, orthogonal to the ADR-0035 decimation** (which shares one broadcast to both encoders and actually *reduces* the contention when hot). **Investigate:** per-side frame-drop accounting under heat; whether merge should tolerate/pad a starved side to keep P/L clip counts aligned; and whether the `MediaFileValidator` drop should be logged louder. Likely couples with the Limiter-2 stagger/shared-encode ideas (deferred as not-worth-the-risk for ~2fps, but the reliability angle is new). *(Source: this session's ADR-0035 device-verify; ffprobe A/B/control in scratchpad.)*

- [ ] **Player → Library transition is jittery/shaky (comfort / a11y)** — **P2 · accessibility**
  Navigating from the video player back to the Library (into the thumbnail grid) produces a jittery/shaky transition the owner reports as headache-inducing. This is the **Compose nav / HWUI path** the profiling already flagged (the "3942 missed vsync / p99 109ms" spike in `docs/superpowers/specs/2026-06-30-dualshot-jitter-thermal-findings.md` §gfxinfo was this post-stop grid-nav, NOT the camera preview). Motion-sensitivity/headache = a real WCAG 2.2 comfort concern (ADR-0020). **Investigate:** reduced-motion gating of the player↔library transition (reuse `ReducedMotion` seam), stabilizing the shared-element / grid entry animation, and a during-nav `dumpsys gfxinfo` to localize the janky frames. *(Source: owner device report 2026-07-01; profiling findings §"Jitter" instrument 3.)*

- [ ] **ThermalStatusSignal fall-dwell can stick elevated on a quiet plateau** — **P3 · signal accuracy**
  ADR-0019's asymmetric hysteresis (`ThermalHysteresis.kt`) is **edge-triggered**: `applyThermalHysteresis` runs only inside the `PowerManager.OnThermalStatusChangedListener` callback, and the 3s fall-dwell steps `stable` **down one level per event**. If the raw status drops and then holds steady (few/no follow-up change events, e.g. an instantaneous cool or a quiet plateau), `state.value` stays elevated until the next `ON_RESUME` `refresh()`. **Device evidence (2026-07-01):** after `cmd thermalservice override-status 3 → reset`, the "device hot" banner **and** the ADR-0035 decimation factor both stayed at SEVERE through a full recording; a forced ON_RESUME (home+relaunch) cleared both immediately. **Safe-direction** (errs cooler / stays lighter), and affects the banner + decimation identically, so **not blocking** — but it under-reports cooldown. **Consider:** a low-frequency periodic re-eval (or a timer that advances the dwell) so the signal converges without needing external events. *(Source: this session's ADR-0035 device-verify.)*
