# ADR-0020: WCAG 2.2 AA by Default

- **Status:** Proposed
- **Date:** 2026-05-29
- **Deciders:** Rova owner (sign-off pending)
- **Supersedes / amends:** none (new cross-cutting policy; interacts with ADR-0007/0013 warning surfaces, ADR-0011/0012 record-home, ADR-0019 thermal hysteresis only insofar as their UI must now meet AA)

## Context

A full WCAG 2.2 Level AA audit (2026-05-29) of all 10 UI surfaces plus two cross-cutting concerns (design tokens, motion) found **161 findings: 3 Blocker, ~58 Serious, ~45 Moderate, ~11 Advisory**. Every surface failed at least one AA criterion. The defects are not random — they concentrate in four root causes:

1. **Low-alpha text tokens.** `RecordChromeTokens.kt` and `SettingsSheetTokens.kt` define ~12 text/label colors at 18–35% alpha over near-black surfaces, yielding 1.5–3.2:1 contrast against the 4.5:1 AA requirement for normal text. Because these are tokens, the failure fans out across record, main-nav, settings, and warning surfaces simultaneously.
2. **Missing screen-reader semantics.** Custom `Modifier.clickable {}` Rows/Surfaces (recovery CTAs, library rows, history warning cards, nav items, snooze chip) lack `role = Role.Button` and `contentDescription`, so TalkBack announces generic containers. Progress and status (loop count, merge progress, FAB Start/Stop) lack `liveRegion`.
3. **No reduced-motion gating.** `RovaAnimations.kt` exports infinite `pulsingOpacity`/`pulsingBorder` primitives with no `Settings.Global.ANIMATOR_DURATION_SCALE` check. Every pulse indicator (recording dot, recovery severity tag, warning snooze chip, background banner) inherits this, failing SC 2.3.3 / 2.2.2.
4. **Undersized touch targets.** Token sizes `camControlSize` (30dp), `stepperButtonSize` (27dp), `navIconBoxSize` (42dp) and several CTA heights (40–46dp) fall below the Material 3 48dp baseline (SC 2.5.8).

The audit artifacts (full report, prioritized backlog, and per-surface findings) are committed under `docs/accessibility/`. Without a standing policy, these regressions will recur as new UI lands, because the project has no invariant that says "UI must be accessible" the way it has invariants for scheduling, export, and recovery.

This project already governs behavioral invariants through ADRs paired with `check*` static-gate tasks wired into `preBuild` (25 such checks today). Accessibility is currently outside that net.

## Decision

1. **All new or changed UI in `:app` MUST meet WCAG 2.2 Level AA.** Concretely, every PR that adds or modifies a composable, RemoteViews layout, or theme/design token must satisfy, for the elements it touches:
   - **1.4.3 Contrast (Minimum):** normal text ≥ 4.5:1, large text ≥ 3:1, against its actual rendered background (alpha-blended over the real surface, not the literal token color).
   - **1.4.11 Non-text Contrast:** UI component boundaries, focus rings, dividers, and meaningful graphics ≥ 3:1. Purely decorative graphics are exempt **only if** explicitly marked `contentDescription = null` (or `importantForAccessibility="no"` for views).
   - **2.5.8 Target Size:** interactive targets ≥ 48×48dp (Material 3 baseline; visual element may be smaller inside a 48dp hit box via `Modifier.minimumInteractiveComponentSize()`).
   - **4.1.2 Name, Role, Value:** every interactive element has an accessible name and a role (`role = Role.Button` etc. on custom clickable containers).
   - **4.1.3 Status Messages:** dynamic progress/status uses `liveRegion`.
   - **2.3.3 / 2.2.2 Motion:** all looping/auto-playing animation is gated on the system reduced-motion preference with a static fallback.
   - **2.4.7 Focus Visible:** keyboard/D-pad-focusable elements show a visible focus indicator (≥ 2dp, ≥ 3:1 contrast).

2. **Contrast and target-size are enforced at the token layer first.** Text/label/divider colors and interactive sizes live in `RecordChromeTokens.kt`, `SettingsSheetTokens.kt`, `RovaTokens.kt`, and the warning token files. Token values MUST be defined to pass AA in their declared use context; per-call-site overrides that drop below AA are forbidden.

3. **A single reduced-motion seam.** A pure helper `shouldReduceMotion(context): Boolean` (reading `Settings.Global.ANIMATOR_DURATION_SCALE`) becomes the one decision point; `pulsingOpacity`/`pulsingBorder` and all callers route through it. This follows the house pure-helper extraction pattern and keeps the seam unit-testable under `isReturnDefaultValues = true`.

4. **Reference artifacts.** The audit deliverables are authoritative remediation inputs and live at:
   - `docs/accessibility/AUDIT_2026-05-29.md` (full report)
   - `docs/accessibility/REMEDIATION_BACKLOG.md` (ranked, deduped backlog)
   - `docs/accessibility/per-surface/*.json` (raw findings)
   New AA decisions amend this ADR; new remediation slices reference the backlog row IDs.

## Consequences

**Positive**
- Accessibility becomes a first-class invariant rather than a one-time cleanup. Token-level fixes (backlog ranks 4–5) clear a large fraction of contrast findings at once with low effort.
- The reduced-motion seam (rank 6) is reused by every future animated indicator, so new pulse UI is compliant by construction.
- Aligns the app with Play Store accessibility expectations and reduces risk of user-facing regressions on TalkBack / Switch Access / external keyboards.

**Negative / costs**
- Raising text alphas changes the visual design slightly (less "ghosted" chrome); the mockup token contract (`RecordChromeTokens.kt`) must be re-baselined with owner sign-off, since `mockups/` is gitignored and specs are inlined.
- Adding semantics/liveRegion/focus modifiers touches many composables; the work is spread across ~35 backlog slices, sequenced Blocker → Serious → Moderate → Advisory.
- Notification live regions on custom RemoteViews are inherently limited by the platform (rank 11 is sized L for this reason).

## `checkA11y*` static-gate suite (1 of 4 built; rest STUB)

**Update 2026-06-03:** `checkA11yAnimationGated` (reduced-motion, §Decision-3) is now **built and wired into `preBuild`** with owner sign-off — see its bullet below. The remaining three stay design-only (NOT built) until their source areas are remediated; landing a gate before its source is GREEN would wire `preBuild` to a permanently-red task.

Consistent with this repo's invariant → `check*` → `preBuild` convention, a future cycle SHOULD add accessibility static checks. **This is a design sketch only and must not be implemented in the current remediation cycle** — landing checks before the source is remediated would wire `preBuild` to a permanently-red gate. Candidate checks, each citing this ADR clause:

- **`checkA11yNoLowAlphaTextToken`** — AST/regex scan of `*Tokens*.kt`: flag any `Color.*.copy(alpha = X)` assigned to a `*Text`/`*Label` token where `X < 0.55f` (heuristic floor below which 4.5:1 is implausible over the dark surfaces). Cites ADR-0020 §Decision-1 (1.4.3).
- **`checkA11yClickableHasRole`** — scan composables: any `Modifier.clickable`/`combinedClickable` on a `Row`/`Box`/`Surface` that is not a Material `Button`/`IconButton` must be accompanied by `semantics { role = Role.Button }` (or an allowlist annotation). Cites §Decision-1 (4.1.2).
- **`checkA11yAnimationGated`** — **BUILT 2026-06-03 (owner sign-off); wired into `preBuild`.** File-level co-presence scan: any `*.kt` under `src/main/java/com/aritr/rova` that uses a RAW infinite-animation primitive (`rememberInfiniteTransition` / `infiniteRepeatable`) must also read the reduced-motion seam (`rememberReduceMotion()` / `ReducedMotion.isReduced`) somewhere in the same file. The self-gating helpers `pulsingOpacity` / `pulsingBorder` are intentionally **not** triggers (they consult the seam internally, so delegated callers are safe). Opt-out hatch: `// a11y-opt-out: <reason>`. Cites §Decision-3 (2.3.3/2.2.2). Design: `docs/superpowers/specs/2026-06-03-checka11y-animation-gated-design.md`.
- **`checkA11yTargetSizeToken`** — flag interactive-size tokens (`camControlSize`, `stepperButtonSize`, `navIconBoxSize`, CTA height tokens) declared `< 48.dp`. Cites §Decision-1 (2.5.8).

When built (a later ADR amendment with owner sign-off — as `checkA11yAnimationGated` was on 2026-06-03), each task is a regex/AST scan wired into `preBuild` via `tasks.matching { it.name == "preBuild" }.configureEach { dependsOn(...) }`, failing with a message that cites the ADR clause — exactly like the existing `check*` gates. Adding any new accessibility invariant follows the same path: amend this ADR's Decision list, add the `checkA11y*` task, wire it into `preBuild`.
