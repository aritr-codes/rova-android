Rova · Player Redesign — "Playback to Standard"

Player Sharing — the share-flow contract

Everything that happens after Share is pressed. player-actions.html defined the Share slot (proposed, trailing) and deferred the flow here. This spec layers on the frozen player-core / player-states / player-gestures / player-actions and inherits every token, layout, state, gesture, and the action framework — it never redraws them, and it does not move Share or redesign Info. Grounded in current source (the reusable primitive is ShareUriResolver.safeShareUri; the player has zero share plumbing today; the playback mediaUri is not a shareable URI) + competitive sharing research — not memory. Preserves ADR-0037/0038/0032/0025/0036, the fail-closed resolver, NO autoplay, and adds no backend capability.

Share layer
00

Model & boundary — what sharing is, and what it must not touch

The one-line model, the backbone it preserves, and the redesign lines it may not cross.
The model in one line

Share = hand the OS a read-only copy of the exact artifact the user just reviewed, via the system chooser, and get out of the way. Rova resolves the shareable URI, sets a temporary read grant, launches the chooser, and owns nothing past the handoff.

It is a route, not a build: the artifact already exists on disk (export ran automatically, or the raw segments were kept). Sharing performs no transcode, no merge, no copy-to-public, no upload. The one reusable primitive — ShareUriResolver.safeShareUri — already backs Library and Preview sharing.

What this spec must NOT do
  • Not move or restyle the Share slot, the Info affordance, transport, timeline, layout, tokens, states, or gestures — those are the four frozen specs.
  • Not invent backend: no new persistence, no new schema field, no cloud/upload/account, no MediaSession, no custom share targets or tracking.
  • Not design editing/trim, DualShot angle-swap, fit/fill, or the Info sheet contents — future specs (§12).
  • Not change the resolver, the export tiers, SAF behavior, the vault privacy model, or the delete transaction.
Frozen · sharing adds NO backend capability

Every step routes something that already exists: the artifact (already exported or kept), the resolution (HistoryArtifactMapper.resolveShareUri logic + safeShareUri), the grant (FLAG_GRANT_READ_URI_PERMISSION, as Library/Preview already set), and the chooser (Intent.createChooser). The player gains exactly one new capability boundary — a Share entry point that resolves from the manifest and calls the existing primitive — and it remains PROPOSED, gated on owner sign-off (the same gate player-actions put on the Share slot itself).

02

Share philosophy — route the existing primitive, own nothing past handoff

First principles for a review player, filtered through "no backend / calm / reversible."

Competitive sharing (Google Photos, Apple Photos, Samsung Gallery, Blackmagic Camera, DJI Mimo) converges on four principles worth keeping and one worth rejecting:

PrincipleRova stance
Share the artifact as-is; don't re-encode inline.Adopt Rova already has the finished file. Share routes it directly — no transcode, no quality picker, no size dialog. Fastest, calmest path.
Hand off to the OS chooser; don't build an in-app target grid.Adopt the system chooser is universal, accessible, and already used by Library/Preview. Rova adds no bespoke share UI.
Temporary, scoped read permission per share.Adopt FLAG_GRANT_READ_URI_PERMISSION on the Intent — the exact grant Library/Preview set. No persisted access.
One clear default; ask only when there is a real choice.Adopt single artifact = zero prompts, straight to chooser. DualShot = the only genuine choice (two files), so the only Rova-owned prompt (§04).
Pro apps: export presets, watermark, resolution ladder, "share to project."Reject these are editor/library features, not review, and all are new backend. Rova shares the recording, not a re-render.
Frozen decision

Sharing is a thin router over an existing primitive. The player resolves the shareable URI from the already-loaded manifest, applies the temporary read grant, and launches the system chooser. It never re-encodes, never copies to a public location it did not already export to, never uploads, and draws no UI past its own optional side-choice sheet. Everything downstream of the chooser is the OS's job.

03

Artifact selection rules — which file is shared, per recording kind

The playback URI is NOT the share URI. Sharing resolves from the manifest, per kind, exactly as the Library already does.
Critical · mediaUri is not shareable

PlayerUiState.Ready.mediaUri is a playback string that, for vault and kept-raw, is a custom sentinel (vaultfile://…, keptsegment://…) that only the in-app resolver understands — an external app cannot read it. Sharing MUST resolve the shareable artifact from the manifest (the same object the resolver already loaded), reusing the Library's existing resolution pipeline — resolveShareUri for the content:// tiers, plus resolveArtifactFile/-Segment + queryMediaStoreUriByPath for the File/Tier2-3 cases, then safeShareUri (resolveShareUri alone returns null for Tier2/3 and kept-raw). It must never pass mediaUri to an ACTION_SEND Intent.

Frozen decision

Share targets the exact identity being reviewed (ADR-0037 — transported, not reconstructed): the merged file for a merged session, this segment for a kept-raw segment, this side for DualShot (§04). If no shareable artifact exists (pre-finalize, or vault), the Share slot is absent, per the present-not-disabled rule from player-actions §05. The resolution reuses the Library's existing manifest→share-URI logic; the player introduces no new resolution path.

04

DualShot sharing — one Rova-owned choice, defaulting to the reviewed side

Two independent output files exist. Choose? Automatic? Both? Grounded in ADR-0037, no invented identity.

A DualShot session has two separate finalized files — portrait and landscape — represented by parallel per-side manifest fields, each addressed by a VideoSide. The player opened one side; its PlaybackIdentity.side is already minted by the Library and transported in. That is the only side whose identity the player holds — but the sibling side's shareable file is a per-side field on the same already-loaded manifest (landscapePendingUri / landscapePublicTargetPath / …), so resolving it reads existing data, it does not reconstruct identity.

OptionVerdictWhy / mechanism
Share the reviewed side (default)Primary pathThe user is watching one angle; sharing it needs zero thought. ACTION_SEND, single URI resolved for identity.side. This is the top, pre-selected option.
Offer the other sideIn the sheetResolve the sibling side from the manifest's per-side field → single-URI ACTION_SEND. Cheap, honest, no new identity.
Share bothIn the sheetACTION_SEND_MULTIPLE with two URIs — the exact Intent Library already uses for multi-item share. Two files, one grant.
Auto-pick without askingRejectSilently choosing (or silently sending both) surprises the user and can leak the angle they didn't mean to send. A real fork deserves a one-tap choice.
A merged side-by-side / stitched exportRejectThat is a new render = new backend + editor feature. Out of scope; belongs to editing if ever.
Frozen decision
  • Single-side sessions: no prompt. Share goes straight to the chooser. The side sheet appears only when the session is DualShot and both sides have a shareable artifact.
  • The side sheet is the one Rova-owned surface in this flow — a lightweight pinned sheet (inherited ink, swipe-down-dismiss), three options, the reviewed side pre-selected. It is not a chooser; picking an option then launches the system chooser.
  • No identity is reconstructed. The reviewed side uses the transported PlaybackIdentity; the other side / both read the manifest's existing per-side fields. If a side has no finalized artifact yet, its option is absent (not disabled).
  • Detailed layout of the side sheet is confirmed here as owned by this spec; its final pixels are transcribed at Compose time from this frozen HTML, like every other surface.

Frozen decision · two URIs, never confused

The playback URI (mediaUri, possibly a sentinel) and the share URI (a real content://) are distinct objects resolved by distinct paths. The FileProvider authority (${packageName}.provider, allowlisted to the videos/ path in file_paths.xml) is the only file-exposure surface — the same one playback uses for kept/vault via its own FileProvider detour.

06

Share preparation — near-instant; there is nothing to build

Because the artifact already exists, "preparing" is at most a URI resolution, not an export.
1 · Trigger
Share pressed. Chrome is up (auto-hide paused on interaction).
→
2 · Fork
DualShot? Yes → side sheet (§04). No → straight on.
→
3 · Resolve
Manifest → share URI (§03/§05). Synchronous for content/FileProvider; a MediaStore _DATA query only for Tier2/3.
→
4 · Launch
Grant + createChooser. System sheet appears.
Why there is no long "preparing" state
  • Merged (Tier1/SAF): the content URI already exists — resolution is a field read. Instant.
  • Kept-raw: FileProvider mint from a known File — synchronous. Instant.
  • Merged (Tier2/3): one MediaStore _DATA query — sub-second, off the main thread. A brief indeterminate indicator is allowed only if it exceeds a frame; otherwise none.
  • No transcode / merge / copy ever happens at share time. Export already ran automatically; sharing routes its output.
Feedback transitions (every one)
StateWhat the user sees
IdleShare slot present (artifact exists). No indicator.
Preparing (Tier2/3 only, if >1 frame)Brief indeterminate spinner on the Share glyph; inherited motion, no blocking modal.
ReadySystem chooser slides up (system-animated).
CancelledChooser dismissed → return to player at the same position. Silent, no toast (nothing failed).
SuccessTarget app takes over. On return, player resumes. No Rova success toast (the target app is the confirmation).
FailureFail-closed toast (§08). Playback untouched.
Frozen decision

Sharing never blocks playback and never shows a heavy progress UI, because there is no heavy work. At most a sub-second resolution. Cancellation and success are silent (the system and target app are the feedback); only failure speaks, and only to say the artifact is unchanged. This matches the calm, review-first product line.

07

System chooser — Rova stops at the handoff

The chooser is the OS's surface. Rova sets the Intent and owns nothing after createChooser.
The Intent Rova builds (identical to Library/Preview)
  • ACTION_SEND for one artifact; ACTION_SEND_MULTIPLE for DualShot "both".
  • type = "video/mp4".
  • EXTRA_STREAM = the resolved content:// URI(s).
  • addFlags(FLAG_GRANT_READ_URI_PERMISSION) — temporary, per-share.
  • launched via Intent.createChooser(...) so the OS always shows the picker.
What Rova does NOT do
  • No custom in-app target grid, no "recent contacts," no per-target shortcuts.
  • No ChooserTargetService, no direct-share tuning, no analytics on which target was picked.
  • No persisted grant, no takePersistableUriPermission. The grant dies with the task.
  • No styling of the sheet — it is drawn dimmed in the specimen precisely to signal "not ours."
Frozen decision

Rova's responsibility ends at createChooser. The chooser's appearance, target list, ordering, and post-pick behavior are the platform's. This keeps sharing universal, accessible by construction, and free of any bespoke surface to maintain — consistent with Library/Preview, which the user may already recognize.

08

Frozen decision · sharing is non-destructive by construction

Every failure path ends in the recording unchanged, playback continuing, and at most one calm toast — never a crash, never a half-sent state, never a mutation. Cancellation is silent (nothing failed). The toast reuses the inherited snackbar idiom and must not overlap the action bar. There is no retry stack and no error dialog; the user simply presses Share again if they wish.

10

Security & privacy — temporary grants, vault never leaves

Temporary permissions, vault behavior, private/FLAG_SECURE recordings, no accidental leakage.
Grant model
  • Per-share, read-only, temporary. FLAG_GRANT_READ_URI_PERMISSION on the Intent — the grant is scoped to the chosen target and revoked by the system when the task ends.
  • No persisted access. No takePersistableUriPermission, no long-lived grant, no re-grant across shares.
  • Single exposure surface. Files reach other apps only through the FileProvider authority allowlisted to the videos/ path; nothing else is exposed.
  • Content URIs (Tier1/SAF) carry the OS's own scoped permission; Rova grants read on them the same way.
Vault & private recordings — the hard line
  • Vault recordings are never shareable from the player. A vaulted session is app-private with "no public copy" and renders under FLAG_SECURE. The Share slot is absent for it — there is no target to resolve, and sharing would defeat the vault's purpose.
  • No accidental leak of the wrong DualShot side — the reviewed side is the default and the fork is explicit; nothing is sent without a tap.
  • FLAG_SECURE is a display protection, not a share gate — but for vault it coincides with "no share slot," so a secure recording is never handed out.
  • If vault sharing is ever wanted, it is a new capability requiring an explicit ADR (a "temporarily de-secure and expose an app-private file" decision) — out of scope here, and deliberately not designed around.
Frozen decision

Sharing grants the minimum access (temporary read on exactly the shared URI[s]) and the vault is categorically excluded — no vault artifact can leave via the player. There is no path by which a private recording is shared without the user explicitly selecting a shareable, non-vault artifact and a target. Any future vault-sharing is gated behind its own ADR, not smuggled in here.

11

Motion & reduced motion — reuse the vocabulary, add nothing

The share flow introduces no new motion language; reduced motion changes animation, never behavior.
Motion (inherited only)
  • Side sheet enters/exits on the inherited container/exit timings (--t-container / --t-exit, --ease-std) and the same swipe-down-dismiss the gesture spec defines. No bespoke transition.
  • System chooser is system-animated; Rova animates nothing for it.
  • Preparing indicator (rare) uses the inherited indeterminate style; no new spinner.
  • Failure toast uses the inherited snackbar entrance. No custom shake/bounce.
  • No new motion token. A step that seems to need one needs a player-core change first.
Reduced motion (inherited invariant)
  • Under rememberReduceMotion(), the side sheet appears without its slide, the toast without its transition — animation disappears, behavior does not.
  • Every option, the chooser, and the toast remain present, in the same place, reachable by the same tap and a11y path.
  • No part of the share flow is discoverable only through motion.
  • Same rule player-gestures/player-actions apply, extended to sharing.
12

Frozen boundary — what this spec owns, what stays elsewhere

Sharing owns the flow after Share is pressed. Nothing leaks into or out of the neighboring specs.
player-sharing.html OWNS
  • The share flow after Share is pressed: trigger → (DualShot fork) → resolve → grant → chooser.
  • The artifact-selection rules per recording kind (§03) and the "resolve from manifest, not mediaUri" contract.
  • The DualShot side-choice sheet (this-side default / other / both) — the one Rova-owned surface in the flow.
  • The URI-ownership map, preparation/feedback transitions, failure handling, and the vault-exclusion + grant security model.
  • The a11y + motion contract for the share flow.
NOT owned here (do not design)
  • The Share slot itself — placement, target size, presence rule — is player-actions.html. This spec only fills it.
  • player-info.html — the Info sheet and any metadata. Untouched.
  • player-editing.html — trim/edit, and any re-render/stitched export (rejected here as new backend).
  • player-dualshot.html — angle swap and DualShot playback controls (this spec touches DualShot only for which file to share).
  • Vault sharing — a future ADR, not a spec, since it is new capability.
  • MediaSession / cloud / accounts / upload — no backend exists; out of scope everywhere.
Boundary rule

This spec may consume the Share slot that player-actions defined and the identity PlayerUriResolver mints, but it may not redefine either. The share-flow contract — resolve-from-manifest, reviewed-side default, temporary-grant, vault-excluded, fail-closed — is frozen here; a change routes back through this HTML and is re-approved first, exactly like every other change in the HTML-first workflow. Sharing remains PROPOSED until the owner signs off the Share slot in player-actions; both gate together.
