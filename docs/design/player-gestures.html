Rova · Player Redesign — "Playback to Standard"

Player Gestures — the interaction contract

Every interaction with the player: single tap, double tap, timeline scrub, play/pause, swipe, long-press, the gesture-conflict priority matrix, auto-hide, and the full accessibility + reduced-motion equivalents. This spec layers on the frozen player-core.html and the frozen-candidate player-states.html — it inherits tokens, layout, controls, states and the a11y floor, and never redraws them. It defines only how a touch becomes an action. Grounded in current source (AutoHideChromePolicy, EdgeSeekZones, SegmentedTimeline, PlaybackSpeedPolicy, PlayerViewModel) + competitive interaction research — not memory. Preserves ADR-0037/0038/0032, the fail-closed resolver, and NO autoplay. No new backend capability; Media3 is pinned 1.4.1 so there is no MediaSession — headset / media-key transport is explicitly out of scope.

Gesture layer
00

Interaction model — principles & boundary

What every gesture must satisfy, and what this spec is NOT allowed to touch.
Seven properties every interaction must satisfy
  • Predictable — same touch, same result, everywhere. No mode-dependent surprises.
  • Low cognitive load — a gesture never requires the user to remember state.
  • Discoverable — nothing load-bearing is gesture-only; every gesture mirrors a visible control.
  • Reversible — every gesture has an obvious undo (seek back, tap again, Back).
  • Consistent — matches platform + gallery convention; no bespoke idioms to learn.
  • Minimal — the fewest gestures that cover the job. Every unused gesture stays unbound.
  • Calm — this is reviewing your own recordings, not consuming a feed. No feed idioms.
The anti-goal
Rova is not a social video feed. We reject vertical-swipe-to-next-clip and hold-to-2× outright (§07/§08): feed-skimming affordances that fight a review posture and collide with the gestures we do keep.
Frozen boundary — inputs this spec builds ON, never edits
  • player-core.html (FROZEN) — layout, controls, chrome policy, tokens, motion vocabulary.
  • player-states.html (FROZEN CANDIDATE) — Loading, Unavailable, Retry, resume/start-over faces.
  • ADR-0037 — playback identity; resolver fails closed; Library sole minter. Read-only.
  • ADR-0038 — lifecycle; fresh player per lease; ownedPlayer() lease-gate; dismiss uses the existing nav-pop → 400 ms deferred release. No new teardown path.
  • ADR-0032 — wall-clock playhead. Scrub reads/writes position through the existing path.
  • Media3 1.4.1 / no MediaSession — no headset, lock-screen, or media-key transport is designed here.
This spec adds NO backend capability
Every gesture resolves to a call that already exists: onSingleTap, onTogglePlay, onSeekRelative, onScrubStart/Update/End, onSetSpeed, nav-pop. Two interactions are proposed additions (boundary haptic §05, swipe-dismiss §07) — both pure UI, no backend, clearly marked and gated behind owner sign-off before Compose.

02

Single tap — reveal chrome

Idempotent reveal + auto-hide re-arm. The one true "show me the controls" gesture.

A single tap anywhere on the video body calls onSingleTap(), which sets chromeVisible = AutoHideChromePolicy.onUserTap(chromeVisible) and bumps interactionTick to re-arm the auto-hide timer (PlayerScreen.kt:301–304). onUserTap always returns true — a tap reveals chrome and never hides it (AutoHideChromePolicy.kt:26–27).

Canonical: tap reveals, auto-hide hides
Hiding is owned solely by the auto-hide timer (while playing) — not by a toggle. This is deliberate and matches the gallery convention (Google Photos, Samsung Gallery): the user never has to guess whether a tap will show or hide. On a paused frame chrome is already pinned, so a tap is a harmless no-op that re-confirms visibility.
  • Priority: a tap that lands on a real control (button, timeline) is consumed by that control; the body-tap sits z-below the chrome (PlayerScreen.kt:333–339). No ambiguity — region ownership (§08).
  • Disambiguation: single-tap fires only after the ~300 ms double-tap window closes (platform detectTapGestures). This latency is inherent to coexisting with double-tap-seek and is accepted industry-wide. See §03 for the short-clip note.
  • A11y: the body carries Role.Button + onClick semantics with player_show_controls_cd (PlayerScreen.kt:353–357) — TalkBack double-tap reveals chrome identically.
03

Double tap — edge seek / center toggle

Adopted. YouTube-family edge-seek, tuned by region. Already shipped (PR #143).

detectTapGestures.onDoubleTap reads the tap x against EdgeSeekZones.zoneFor(offset.x, surfaceWidthPx) (PlayerScreen.kt:343–350): left third → onSeekRelative(−10 s), right third → onSeekRelative(+10 s), center third → onTogglePlay(); then onSingleTap() reveals chrome. Zone split is leftFraction = rightFraction = 1f/3f; degenerate width or negative x falls back to TOGGLE (EdgeSeekZones.kt:17–35). Seek distance SEEK_DELTA_MS = 10_000L.

Accepted interaction (unchanged from shipped)
Edge-thirds double-tap seek + center-toggle is kept verbatim. Chained taps accumulate and are reversible; each jump reveals chrome so the position lands in view. Center-third double-tap duplicates the play/pause button — an intentional convenience shortcut, consistent with Google Photos.
Open question (owner sign-off — NOT changed here)
Research (Google Photos) uses a 5 s hop on short clips where a 10 s jump overshoots, and disables the gesture below a minimum clip length. Rova ships ±10 s today. This spec does not silently change it: the constant lives in SEEK_DELTA_MS and is owner-frozen. Flagged as a candidate — either a shorter hop or a length-gated disable — for a future decision. If double-tap-seek were ever disabled on very short clips, the single-tap chrome reveal could become instant there (no double-tap window to wait on) — a predictability win.
04

Timeline — the spine of playback

Tap-to-seek + drag-scrub + clip-boundary snap. Already the richest control; kept as the anchor.
Current behavior (kept)
  • Tap-to-seek: detectTapGestures → onSeek(positionFromFraction(x/width)), clamped 0..1 (SegmentedTimeline.kt:127–132).
  • Drag-scrub: a separate detectDragGestures block; onDragStart → onScrubStart(), per-move change.consume() + onScrubUpdate, onDragEnd/Cancel → onScrubEnd (:133–164).
  • Clip-boundary snap: snapIfNear within snapWindowMs = (totalMs/50).coerceIn(150, 1500) — magnetically catches segment edges (:118, SegmentedTimelineMath.kt:141–147). Segment ticks are drawn at cumulative offsets.
  • Coalescing: scrub-update emits only when the snapped target changes (:159–162) — no flooding the player with sub-pixel seeks.
  • Edge behavior: fractions clamp to 0..1; position clamps to 0..total. No wrap, and an over-drag on the bar never triggers dismiss (§07 region ownership).
Evaluated additions — reasoned decisions
  • Boundary haptic — RECOMMEND (proposed, no backend). A single performHapticFeedback (CLOCK_TICK) as a scrub crosses a clip boundary / hits 0:00 or end. Reinforces "reversible, predictable"; composes with the existing snap; no dependency. Reduced-motion / system-haptics-off respected.
  • Vertical fine-scrub — DEFER (reject for now). Drag-away-to-slow-precision is premium but (a) needs a second axis on the same pointer, which directly collides with swipe-dismiss (§07), and (b) segment snap already gives frame-adjacent precision on short recordings. YAGNI until users demand it.
  • Thumbnail scrub-preview — OUT (backend capability). A frame preview above the thumb needs frame-at-time retrieval Media3 1.4.1 does not give us for free — that is new backend capability, forbidden here. Recorded as a future spec item, not designed.
A11y — the timeline is already a real slider
The gesture box exposes progressBarRangeInfo, stateDescription (segment n/n + mm:ss), setProgress (TalkBack swipe-up/down + keyboard adjust), and two CustomAccessibilityActions (prev/next segment), plus a polite liveRegion announcing segment changes (SegmentedTimeline.kt:165–181, 239–254). This is the accessible seek path — no gesture-only scrubbing.
05

Play / Pause & speed

Dedicated button primary, center-tap shortcut secondary. State owned by the ViewModel.
Behavior (kept)
  • Primary: center 48 dp Surface(onClick = onTogglePlay) in ControlsRow (PlayerScreen.kt:674–689) — labelled, Role.Button, always the accessible path.
  • Shortcut: center-third double-tap also toggles (§03). A convenience, never the contract.
  • Skip: Replay-10 / Forward-10 IconButtons mirror the double-tap edge-seek for TalkBack + discoverability (:667–696).
  • Speed: the PR-7 chip cycles 0.5× / 1× / 1.5× / 2× via PlaybackSpeedPolicy.next (session-only, PlaybackSpeedPolicy.kt:23–24), applied through setPlaybackSpeed gated on COMMAND_SET_SPEED_AND_PITCH. Explicit + reversible.
State ownership

PlayerViewModel.togglePlayPause() owns transport (PlayerViewModel.kt:306–310): pause persists position, play resumes; isPlaying flows out via PlaybackProgress. The gesture layer never mutates ExoPlayer directly — it calls the VM, which is lease-gated by ownedPlayer() (ADR-0038).

Out of scope — no backend
No MediaSession (Media3 1.4.1, deliberate: PlayerScreen.kt:156). Therefore no headset button, no Bluetooth transport, no lock-screen / notification controls, no hardware media keys. This is a known gap, not a gesture this spec may design — it requires backend capability we are forbidden to invent. Keyboard Enter/DPAD-center still activates the focused on-screen button (§10).
06

Swipe — evaluate from first principles

Swipe-down-to-dismiss: adopt (proposed). Vertical-swipe-to-next: reject. Edge-swipe: system.
Swipe-down-to-dismiss — ADOPT (proposed, owner sign-off before Compose)

The reversible, expected gallery exit (Google Photos / Apple Photos / Samsung Gallery all do it). A vertical drag that begins on the media body rubber-bands the frame with the finger; releasing past --dismiss-commit (≈120 dp) or a downward fling ≥ --dismiss-velocity commits the dismiss, otherwise it springs back. Commit routes the existing nav-pop — the same path as Back — so ADR-0038 teardown (park → 400 ms deferred release) is unchanged. No new lifecycle, no new backend.

  • Not present today (no swipe/drag-to-dismiss found in PlayerScreen.kt). This is a genuine addition — hence proposed + gated, not asserted as existing.
  • Accidental-dismissal mitigations: generous distance threshold; interactive rubber-band (you see it coming and can abort by dragging back); directional axis-lock in the first few px; velocity shortcut only on a deliberate fling.
Conflict with the timeline — resolved by region ownership
A vertical drag that starts on the timeline or a control belongs to that control (scrub / no-op), never to dismiss (§08 priority matrix). Because Rova defers vertical fine-scrub (§04), a vertical drag on the media body is unambiguous — it can only mean dismiss. Keeping fine-scrub out is what makes swipe-dismiss safe.
Vertical-swipe-to-next-clip — REJECT
Reels/TikTok map vertical swipe to next/previous item. That is feed navigation: it fights swipe-dismiss head-on, it fights fine-scrub, and it is the opposite of a calm review posture. Rejected outright. Moving between recordings happens in the Library, not by swiping inside the player. Edge-swipe stays the OS system-back gesture (which already routes the same nav-pop) — this spec does not intercept it.
07

Long press — evaluate, then reject

Unbound on the player surface. Reserved.

Long-press is unbound today (NOT FOUND in PlayerScreen.kt). The only common player use is hold-to-2× (Reels/TikTok/YouTube) — a feed-skimming affordance: undiscoverable, un-calm, and useless for frame-accurate review of your own footage. Photos/Gallery reserve long-press for Motion-photo / selection, not playback.

REJECT — leave unbound (YAGNI)
Speed already has an explicit, reversible chip (§05); a hidden hold gesture would duplicate it worse. Leaving long-press unbound keeps it free for a future system gesture (e.g. share / scrub-lock) without retraining users, and removes one more thing that can fire by accident during a calm review. No accessibility cost — nothing is lost because nothing was bound.
08

Gesture-conflict priority matrix

The most important section. Nothing is ambiguous. Resolution order is fixed.

Every raw touch resolves top-down. The first rule that matches wins; it never reclassifies.

#RuleWins forLoses / deferredMechanism
1Region ownership — where the touch begins Controls & timeline own their gestures (button click, scrub) Body-only gestures (tap-chrome, double-tap-seek, swipe-dismiss) never fire from a control z-order: chrome above body; timeline drag change.consume() (SegmentedTimeline.kt:150)
2Axis lock — first move vector past slop classifies drag axis Strong-vertical on body → swipe-dismiss; horizontal on bar → scrub A locked axis never converts (a scrub can't become a dismiss) directional lock in first few px (proposed dismiss detector); scrub already horizontal-only
3Tap vs drag — movement past touch-slop before release Past slop ⇒ drag/scrubWithin slop ⇒ tap platform ViewConfiguration touch-slop (~8dp)
4Single vs double tap — second tap within the window 2nd tap ⇒ double-tap-seek/toggleNo 2nd tap ⇒ single-tap-chrome (fires after the window) detectTapGestures ~300 ms window (PlayerScreen.kt:328)
5TalkBack override — touch-exploration active Semantics tree (focus + activate + slider actions)ALL raw gestures suspended; auto-hide off isTouchExplorationEnabled read once (PlayerScreen.kt:271–275, 284)
Every pairwise conflict, explicitly
  • Timeline drag vs body swipe-dismiss → Rule 1 (region: drag started on the bar) then Rule 2 (axis).
  • Single-tap vs double-tap → Rule 4 (window). Single-tap accepts the ~300 ms latency by design.
  • Double-tap-seek vs swipe-dismiss → Rule 3 (a tap has no slop travel) — a double-tap is two taps, never a drag.
  • Button press vs body tap → Rule 1 (the button consumes; body-tap is z-below).
  • Any gesture vs TalkBack → Rule 5 wins unconditionally.
09

Auto-hide interaction

Builds directly on AutoHideChromePolicy. Timing is frozen — 3 s — and NOT redesigned.

The policy is pure: shouldRunHideTimer(isPlaying, isScrubbing, chromeVisible, speedMenuOpen) = isPlaying && !isScrubbing && chromeVisible && !speedMenuOpen (AutoHideChromePolicy.kt:19–24). The consumer runs a delay(3000); chromeVisible = false effect keyed on (isPlaying, isScrubbing, chromeVisible, interactionTick) (PlayerScreen.kt:283–292). This spec defines the interaction contract over that timer; it does not touch the 3 s value.

Contract
Chrome hides only by this timer, and only while playing, idle, non-scrubbing, non-TalkBack. Every other state pins it. No gesture hides chrome directly (§02).

The rule that makes the minimal aesthetic safe
The visible skip / play / Back / slider controls are the accessibility layer. Gestures are shortcuts over them, never replacements. The minimal look may never delete a visible control — doing so would strand every non-gesture input path. Announcements: the timeline's polite liveRegion speaks segment changes; state changes (Loading/Ready/Unavailable) are announced by player-states §08, not re-specified here.
11

Motion & reduced motion

Reuse player-core's vocabulary. No new motion language. Reduced motion strips animation, keeps behavior.
Motion — inherited only
  • Chrome show/hide: fadeIn/fadeOut via AnimatedVisibility (PlayerScreen.kt:400–403, 477–482). No new easing.
  • Timeline fill: RovaMotion.containerSpring(), snapping during active scrub (SegmentedTimeline.kt:191–199).
  • Double-tap feedback: if a seek arc/ripple is added, it uses --ease-std + --t-micro/--t-small from FAMILY 3 — no bespoke curve.
  • Swipe-dismiss (proposed): the frame follows the finger 1:1 (no easing while dragging); spring-back uses containerSpring(); commit hands off to the nav transition.
Reduced motion — behavior survives, animation dies
  • Gate: rememberReduceMotion() (PlayerScreen.kt:264, ReducedMotion.kt:46).
  • Chrome appears/disappears instantly (EnterTransition.None) — the reveal still happens.
  • Timeline fill snaps to position — the seek still lands.
  • Double-tap seek: the jump is instant with no arc; the ±10 s still applies.
  • Swipe-dismiss: still tracks the finger (that's function, not decoration); only spring-back easing is dropped.
  • Boundary haptic (proposed) is not motion — it stays, gated separately by system haptics settings.
Invariant
Reduced motion changes how a gesture animates, never whether it works or what it does. Every interaction in §02–§09 is fully usable with all animation removed.
