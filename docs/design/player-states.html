Rova · Player Redesign — "Playback to Standard"

Player States — before, during & after playback

Every state the player passes through: Loading, first-frame handoff, the resume / start-over affordance, every Unavailable cause, Retry, and the runtime Ready→Unavailable flip — each with accessibility announcements, reduced-motion behavior, and motion sequencing. These are primary journeys, not edge cases: the resolver fails closed (ADR-0037), so Unavailable and Loading are common. This spec layers on the frozen player-core.html — it inherits the token model, three-zone layout, band substrate, immersive policy and a11y floor, and never redraws them. Grounded in rova-player-discovery.md + current source (PlayerUiState / PlayerUriResolver / ResumePolicy) — not memory. Preserves ADR-0037/0038/0032, the fail-closed resolver, and the NO-autoplay trust rule. No new backend capability — Retry re-runs the existing resolve / attach / prepare path.

Specimen state
00

State model — the honest map

Three UI states in code today; this spec designs all their faces + transitions.
What the code actually exposes

PlayerUiState (PlayerUiState.kt:25) is a sealed interface with exactly three cases: Loading (:26, the construction default), Ready (:28), and Unavailable(reason: UiText) (:60). There is no cause enum — every distinct failure is encoded only by which string resource the resolver passes into that one UiText.

So the design job is: (1) give Loading a real, poster-first face; (2) give each of the six Unavailable causes (8 resolver returns + 2 runtime failures) a designed frame that reads its cause and offers the right action; (3) design the three transitions — entry→Loading→Ready, Loading→Unavailable, and the mid-session Ready→Unavailable flip.

Design principles for states (inherited + specific)
  • Every state is designed. Loading and Unavailable are common (fail-closed resolver) — never a footnote.
  • Poster-first, never black. Show the warm entry poster immediately; escalate to a loading cue only after a grace delay.
  • Transient offers Retry; permanent does not. A "still exporting" clip gets a way forward; a deleted file does not pretend to.
  • Calm, non-blaming copy. Plain language + reassurance, never a raw exception or code.
  • Failure is reachable mid-playback. A codec error freezes the frame and fades an overlay in — it never hard-cuts to black (RK7).
  • Announce state changes. Loading/Ready polite, failure assertive — a screen-reader user is never left in silence.
  • Nothing means anything by motion. The spinner, the poster cross, the overlay fade all have static equivalents.
Boundary with core — this spec does NOT reopen the shell
Token families, the three-zone layout, immersive chrome policy, and control/timeline placement are frozen in player-core.html. States reuses them verbatim and adds only the state faces (loading cue, error card, resume pill) and the state machine. It introduces no new colour family — only a handful of state-timing/geometry tokens (§02).

Rules
  • Never open on a bare spinner. If a poster exists, it IS the loading backdrop; the cue (if any) sits over it.
  • Grace before any cue (--loading-grace): sub-400ms resolves show no spinner at all — no flash-of-loading.
  • Swap on the callback, not a timer — the 1500ms --poster-ceiling is a safety cap for a slow/absent first-frame, not the normal path.
  • R6 — extend the poster handoff to all entries (or a graceful non-poster fallback = the grace-delayed cue). Vault/legacy/deep-link should not flash a blacker entry than the identical Library open.
  • Loading is a Box over the video surface; the surface (TextureView) is already inflated so the poster composites over the shutter (no HW-overlay flash).
Today → target

Today: Loading = a single centered Text("Loading…") at white .7, no spinner, no poster on the cold path (PlayerScreen.kt:209). A slow resolve looks frozen.

Target: poster-first on warm; grace-gated token spinner on cold; both announce politely. The warm path is silent and instant; the cold path degrades gracefully instead of looking hung.

No autoplay is untouched
First-frame handoff renders the first frame; it does not start playback across items. Press-to-play and the resume policy (below) own whether/where the playhead sits. The loading design never introduces autoplay.
04

Resume & start-over

Silent resume + a quiet, non-modal cue. Exact-slot semantics preserved (ResumePolicy).
Behavior (grounded in ResumePolicy.kt)
  • Silent resume is the default — no interstitial dialog. Rova's clips are short, self-recorded; a modal "Resume / Start over" gate is streaming-grammar, wrong here (competitive: gallery apps resume silently).
  • Near-end auto-resets to 0 already: resolveOpenPosition returns 0 when the saved point is within nearEndThresholdMs = min(3000, max(1000, dur×0.02)). So the cue only appears when the playhead is genuinely mid-clip.
  • The cue: a non-modal pill — "Resuming · m:ss" + a Start over peer — self-dismisses at --resume-cue-dwell (4s) or on first interaction. It makes the silent restore visible (R5), nothing more.
  • Start over seeks to 0 and rewrites the same exact resume slot (#seg<N> / named side) — no bleed across segments/sides (ADR-0037 slot semantics).
  • At/after end parks at durationMs (existing behavior) — the cue reads "Start over" as the sole path.
What this spec does NOT change
  • The resume math (ResumePolicy/PlayerResumeMath) is read-only backbone — the cue only reads the resolved start position.
  • Whether entry auto-plays or waits for press is the core/gesture play policy — states only adds the visible cue + Start-over. It never adds cross-item autoplay.
  • The slot key is minted/threaded by the Library + resume seam; the cue does not re-mint identity.
Frozen — no modal resume gate
Resume stays silent + visible, never a blocking choice dialog. Any future change to a modal gate must route back through this HTML.

Transient vs permanent — how the UI decides (no resolver change)

The resolver carries only a UiText reason — no severity. So the action is chosen by a pure presentation classifier keyed on the string-resource id: PlayerUnavailableAction.of(@StringRes reason) → {RETRY, DISMISS}, exhaustive over the six known ids, unknown → DISMISS (fail-safe: never offer a retry we can't honor). This mirrors the M9 MergeFailureReason precedent (closed set, message-free) and touches no backend — the resolver / ADR-0037 matrix stays read-only.

Cleaner long-term: a typed reason on Unavailable classified at the resolver seam — but that amends ADR-0037's contract, so it's deferred; V1 uses the string-keyed classifier.

Frozen — the triad & the neutral glyph
Every Unavailable frame is neutral glyph + one plain line + the right action(s) — no raw exception, no error code. The glyph is neutral (--media-ink-dim on a hairline ring), never the trust severity red: an unopenable recording is an absence, not a hard-block warning. Copy is calm and specific per cause. This frame is the single face for both entry-time and mid-session failure (§06).
06

Runtime Ready→Unavailable flip (RK7)

An already-playing clip can fail. It must degrade gracefully, not hard-cut to black.
Sequence
  • Codec/container error fires onPlayerError → VM writes Unavailable(playback_failed) (:160) while the surface still shows the last decoded frame.
  • Freeze on that last frame — do not clear the surface to black. Apply a light .34 dim so the card reads.
  • Fade the error card in over the frozen frame (--t-small 200ms, --ease-std). Reduce-motion → appears instantly, no fade.
  • Retry resumes from the failed position (re-prepare), not from 0 — the user keeps their place.
  • The frame reached mid-session is the same card as entry-time failure — one design, two reach-points.
Why not black

A hard cut to a black error screen mid-playback reads as a crash and discards the visual context of where the failure happened. Freezing the frame + dimming keeps continuity; the overlay says "this specific moment hit a problem, here's your way back." (Competitive: freeze-last-good-frame, overlay, retry-from-position.)

Design the flip as a first-class path
A frame that only anticipates entry failure will still jar mid-session. This spec makes the Unavailable frame reachable from a live frame, transitioning in — not just from Loading.

Focus, targets, motion
  • Retry / Back are focusable, labeled, ≥48dp targets. On an Unavailable frame, focus lands on the primary action.
  • The poster carries a content description (or is marked decorative if the title already conveys it).
  • Reduce-motion: spinner → static ring; poster cross-fade → hard cut; card fade → instant. No state cue is motion-only.
  • Touch-exploration state re-arms on toggle (today it's read once at composition — a discovery gap to fix in the states transcription).
  • Auto-hide stays TalkBack-suppressed (inherited from core) — an Unavailable card never auto-hides while focused.

Implementation notes (for the later Compose transcription)

Non-binding until freeze; captures the transcription intent. No backend touched.
  • Loading: replace the single Text (PlayerScreen.kt:209) with a poster-first Box: poster if entryPoster != null, else a grace-gated (--loading-grace) token spinner + label. Spinner is reduce-motion-gated.
  • Unavailable: replace the Back-only card (PlayerScreen.kt:220) with the triad card + a pure PlayerUnavailableAction.of(@StringRes) classifier deciding Retry-vs-Back. Classifier is a pure JVM-tested helper (M9 precedent) — no resolver change.
  • Retry: add a retry() on PlayerViewModel that re-enters the existing resolve/attach path (not_finished/init_failed) or calls ownedPlayer()?.prepare() at last position (playback_failed). No new lease/engine capability (ADR-0038 unchanged).
  • Runtime flip: on onPlayerError, do not clear the surface — render the card over the frozen frame with a .34 dim. Freeze is the absence of clearVideoSurface until Back/Retry.
  • Resume cue: a non-modal pill shown when the resolved start position > 0 (read from the existing PlayerResumeMath result); Start over seeks 0 + rewrites the same slot. No new persistence.
  • A11y: add a state liveRegion (polite Loading/Ready, assertive Unavailable) + resume state on the Play label. Re-arm touch-exploration on toggle.
  • R6: extend PlayerPosterHandoff.set to Vault/legacy/deep-link entries (or accept the grace-gated cold cue as the graceful fallback).
  • Strings: add the §09 strings to values/ + values-es/ together.
  • Tests / gate: the classifier + any contrast pin go into the existing pure-JVM test family — no new gate (protection = pure helper tests + the core scrim-composite pin, per the ADR-0036/0038 precedent). Card ink pairs ride the core over-media contrast pin (§07 of player-core); a solid --pin-surface card is the easy case (opaque, already Trust-swept).
  • Backbone read-only: PlayerUriResolver (ADR-0037), PlayerEngine/Ledger (ADR-0038), ResumePolicy/WallClockTimeline (ADR-0032) are untouched. This spec is chrome + presentation only.
