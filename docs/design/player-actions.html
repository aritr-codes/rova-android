Rova · Player Redesign — "Playback to Standard"

Player Actions — the action-architecture contract

WHAT actions exist in the player and HOW they are presented — the slot framework every later player spec plugs into instead of inventing its own controls. This spec layers on the frozen player-core.html, the frozen-candidate player-states.html, and the frozen player-gestures.html — it inherits tokens, layout, states, and the entire gesture contract, and never redraws them. It defines primary action, secondary actions, overflow philosophy, context-sensitive visibility, the destructive-action policy, and where metadata lives. Grounded in current source (the player has zero file-level actions today — Back + transport + timeline only; no Share, Delete, Info surface, or overflow) + competitive action-architecture research — not memory. Preserves ADR-0037/0038/0032, the fail-closed resolver, NO autoplay, and adds no backend capability.

Action layer
00

Action model — principles & boundary

What every action must satisfy, and what this spec is not allowed to touch.
Seven properties every action must satisfy
  • Calm — the player is for confidently reviewing your own recordings, not managing a media library. Management verbs belong in the Library.
  • Minimal — an action earns a slot only if it is a genuine review verb and uses capability that already exists. Empty is a valid state.
  • Obvious — every action is a visible, labelled control. Nothing load-bearing is gesture-only (inherited from player-gestures).
  • Review-first — playback/scrub is the true primary; file-level verbs are secondary and few.
  • Reversible — no action silently destroys or mutates. Anything destructive is edge-placed, confirmed or undoable, and never in the player at all unless it must be.
  • Consistent — matches the gallery convention (Share/Info as peers, delete edge-placed) so nothing new must be learned.
  • Low cognitive load — a fixed number of slots in fixed positions; the set never rearranges by mode.
What this spec must NOT do
  • Redesign controls, loading, unavailable, resume, tokens, layout, typography, or visual hierarchy — those are player-core / player-states.
  • Redesign any gesture, auto-hide, or the priority matrix — that is frozen player-gestures.
  • Design the sharing flow, editing/trim, DualShot angle-swap, fit/fill, or the full info sheet contents — those are future specs (§13).
  • Invent backend capability — no new persistence, no new schema field, no favorite/reviewed/archive flag, no MediaSession.
Frozen · this spec adds NO backend capability

Every action defined here either (a) is already shipped (transport, timeline), (b) routes an existing capability (ShareUriResolver.safeShareUri, already used by Library/Preview), or (c) is pure presentation of already-loaded data (a details sheet over the manifest the resolver already produced). No action here introduces a new VM method that persists, deletes, or mutates state that does not already have a code path. Two proposals (Share entry point, Info affordance) are marked PROPOSED and gated behind owner sign-off before any Compose work — exactly as player-gestures gated its haptic and swipe-dismiss proposals.

02

Primary action — playback is the primary; there is no hero CTA

Is there a single bottom-anchored primary action? Should there be? When does it disappear?

The primary action is the transport cluster — the shipped bottom-anchored −10 / Play-Pause / +10 controls plus the timeline spine (ControlsRow, SegmentedTimeline). It is the single dominant on-media action, it is already frozen in player-core, and this spec adds nothing above it. The convergent gallery principle holds: in a review player the "primary-ness" belongs to playback, not to any file-level toolbar button. The media is the content; chrome recedes.

Frozen decision
  • Exactly one primary cluster, always the transport. No Share, Save, or Export button is ever elevated to a hero/FAB. Adding a second primary would compete with playback and break "review-first / calm."
  • It never changes identity by context. Play/Pause is Play/Pause everywhere; it does not morph into Retry, Share, or Merge. (State-specific actions like Retry live in their own state face per player-states, not on the transport.)
  • When it disappears: only in Loading (no player yet) and Unavailable (nothing to play) — both defined by player-states. In every playable state the primary is present. It hides with the chrome under the auto-hide timer (inherited player-gestures) but is one tap away.
03

Secondary actions — the two-slot ceiling

Determine the permanent secondary actions. Every one must justify its existence.

The gallery convergence is Share + Favorite + Delete + Info, capped at ~4–5 before overflow. Rova is stricter: an action earns a permanent slot only if it is a genuine review verb AND uses capability that already exists (no invented backend) AND is not a library-management verb. Applying that filter to the full candidate list:

Frozen decision · the secondary set is at most two, and both are proposals

The permanent secondary set is Info (leading) and Share (trailing) — a hard ceiling of two file-level verbs. This is deliberately below the gallery's 4–5: Rova is review-first, and Delete/Favorite/Archive (the actions that push galleries to 4–5) are all Library-owned here. Two peers, fixed positions, never rearranged.

Both are PROPOSED, not shipped. Neither exists in the player today. They are recommended additions gated behind owner sign-off before Compose, on the same discipline player-gestures used. If the owner declines either, the framework still stands — the slot simply stays empty, which §04 already treats as valid.

Open question for the owner — deferred, not silently decided

The no-new-features rule says a new feature must replace one or solve a demonstrated usability problem. Info and Share are the two most defensible additions, but they are still additions. This spec recommends them and proves they add no backend; it does not assume the owner wants them. Sign-off is required before either is transcribed to Compose. Until then the player ships as today: transport + timeline + Back only.

04

Overflow philosophy — the slot exists, Rova populates nothing

What belongs on-screen, what belongs behind overflow, what never belongs in the player.
The rule (inherited from gallery convention)

Frequency × reversibility. The few most-used, safe-or-common actions stay on-bar; everything rare or system-adjacent goes behind "…". Apply it to Rova and the on-bar set is already tiny (Info, Share), and the "rare but real" bucket is empty — there is no Cast, Print, Details-export, or Slideshow capability to demote.

So Rova ships no overflow menu today. The framework names the slot so future specs (editing, DualShot) have a defined home for a rare action instead of inventing a new one — but an empty overflow is the correct current state, not a gap. A "…" with one item is worse than no "…".

Three-tier placement contract
TierRova today
On-media (always)Transport + timeline (primary). Info + Share (proposed peers).
Overflow "…"Empty reserved; no action qualifies. Future editing/DualShot specs may claim it — via their own HTML, re-approved first.
Never in the playerCast · Print · Set-as-wallpaper · Slideshow · library organization (albums/move) · engagement (likes/comments/counts). Not review actions; several have no capability at all.
05

Context-sensitive visibility — slots appear only where meaningful

Action visibility across every state. Slots are present-or-absent, never disabled-but-visible.

Actions bind to state (from player-states) and to artifact shape (from ADR-0037 identity). The governing rule: a slot is shown only when its action is possible and meaningful. Rova never renders a greyed-out action — an absent slot is calmer and unambiguous.

State / shapePrimaryInfoShareNotes
Loading———No player, no manifest resolved. Back only.
Ready · playing✓✓✓Full set (chrome auto-hides per player-gestures).

Finished (ended)✓✓✓Play restarts; no distinct "finished" action set (no autoplay, no "next").

Frozen decision

Visibility is by presence, not by enabled-state. If an action cannot run in a state, its slot is removed, not disabled. The transport is present in every playable state and absent only in Loading/Unavailable. Info and Share ride with the artifact: present whenever there is a resolved artifact to describe or send, absent otherwise. DualShot keeps the Share slot present but defers how many sides to the sharing spec.

06

Selection & mutation philosophy — the player reviews one artifact

Should actions mutate immediately? Need confirmation? Sheets? Undo?

The player is fundamentally reviewing a single artifact — there is no selection model, no multi-select, no batch. From first principles:

Mutation rules
  • No player action mutates the artifact in place. Transport, scrub, and speed are ephemeral view state (speed is session-only). Info is read-only. Share hands a copy to the OS; it does not alter the recording.
  • Nothing needs undo inside the player, because nothing destructive lives inside the player. The reversibility burden is met by not placing destructive actions here (§07).
  • Share opens a system surface (the OS share sheet), which is itself cancellable — the user can always back out before anything leaves the device.
  • Info opens a sheet, not a screen — a review glance, dismissed with a swipe or Back, returning to the exact playback position.
First-principles conclusion

Because the player holds one artifact and performs no in-place mutation, it needs no confirmation dialogs, no undo stack, and no selection UI. Confirmation and undo become relevant only for destructive actions — and Rova's answer is to keep those in the Library (§07), where the atomic delete-then-discard transaction and the grid-level undo already live. The player stays a calm read-and-send surface.

07

Destructive-action policy — defined here, placed in the Library

Delete / Discard / Remove: placement, hierarchy, confirmation, undo, relationship to the Library.

Galleries converge on delete-in-the-detail-view (edge-placed trash + soft-delete + Undo). Rova diverges deliberately: the product line is "confidently review, not manage a media library," and deletion is an atomic Library transaction (ADR-0036 delete-then-discard, per-session, batch-aware). A single-item player delete would have to re-enter that transaction and mutate the grid the user came from — coupling the calm review surface to library-management state. So:

Frozen decision · destruction is Library-owned
  • No Delete / Discard / Remove in the player. The one place to delete a recording is the Library, where the atomic transaction and undo already exist.
  • Back is not destructive — leaving the player never deletes; it routes the existing nav-pop (ADR-0038 unchanged, from player-gestures).
Policy · binding on any future spec that DOES add destruction to a Rova surface

If the owner ever decides a destructive action must live in the player, it MUST follow this policy (defined once, here, so future specs comply): (1) Placement — edge of the chrome, never adjacent to Play/Pause or Share; visually separated from the safe cluster. (2) Hierarchy — never the primary; never a hero. (3) Confirmation/undo — because Rova's delete IS recoverable via the ADR-0036 transaction, prefer immediate + Undo snackbar over a blocking dialog — but the snackbar must not overlap the action bar (a documented gallery friction). (4) Library relationship — one shared deletion transaction with the grid; on delete the player pops back and the item is gone from the grid atomically; no "deleted but still shown" state. This is an open owner question, recommended default = keep it Library-only.

08

Review actions — which belong in the player, which in the Library

Keep · Discard · Archive · Move to Vault · Mark reviewed · Hide.

These read like "review" verbs but are all organization / lifecycle verbs — they change how the artifact is stored or categorized, not how it is watched. The player watches; the Library organizes.

VerbHomeReason
Keep / DiscardLibrary / RecoveryThese are recovery-classification outcomes (RecoveryCard: Merge / Keep / Discard), not playback. They already live on the Recovery surface.
Archive / HideLibrary no capabilityNo archive/hidden state exists in the schema. Adding one = new backend + new management concept. Rejected outright.
Mark reviewed— no capabilityNo "reviewed" flag exists. The act of watching is the review; a manual toggle is a feature with no demonstrated need.
Move to VaultLibraryVaultMover capability exists but vault membership is a storage move. In the player, vault status is passive (FLAG_SECURE only, isVaulted). Moving in/out is a Library action.
Favorite— no capabilityNo favorite flag. New backend + organization verb. Rejected.
Rejected from the player — every one

None of these are review actions and most have no backing capability. Placing them in the player would (a) invent backend, (b) turn the calm review surface into a management console, and (c) duplicate verbs that already have a correct home (Recovery/Library). The player's only lifecycle relationship to the artifact is read it (Info) and send it (Share).

09

Metadata actions — one Info affordance, a details sheet on demand

Should technical detail be shown? Where?

The universal gallery rule: expose the 1–2 review-relevant facts inline, gate the rest behind one affordance (Apple's ⓘ + swipe-up sheet; Google's details panel). Rova already shows the inline facts — the top bar carries date-time and clip count, and the readout carries wall-clock time-of-day and gap (all display-only, shipped). What is missing is the "tell me more" affordance.

Frozen decision
  • One entry point: the Info ⓘ slot (leading, top-trailing). No metadata is ever added permanently over the frame — the inline facts already present are the ceiling for always-on text.
  • It opens a sheet, not a screen — a glance that returns to the exact position. Consistent with the inherited swipe-down-dismiss idiom (player-gestures): a light-weight overlay, not a navigation.
  • Contents = already-loaded manifest data only. No probe, no new read, no backend. If the sheet ever needs richer technical data (codec/bitrate), that is a new capability and belongs to a future player-info.html, re-approved first.
  • This spec defines only the entry point + that a sheet exists, not its full layout. The sheet's visual design is a future spec's job.

11

Motion — reuse the player-core vocabulary, add nothing

Action appearance and feedback use the inherited motion ladder only.
  • Slot reveal/hide rides the existing chrome fade — actions appear and disappear with the top bar and bottom panel under the auto-hide timer. No per-action entrance animation.
  • Press feedback is the inherited glyph-fill press state (--glyph-fill → --glyph-fill-press), same as every shipped control. No ripple redesign, no bespoke bounce.
  • Info sheet uses the inherited container/exit timings (--t-container / --t-exit, --ease-std) and the same swipe-down-dismiss motion the gesture spec already defines for the player.
  • Share hands off to the OS immediately; the system animates its own sheet. Rova animates nothing for it.
  • No new motion language. An action that seems to need one needs a design change in player-core first.
12

Reduced motion — actions appear instantly, behavior identical

Reuse the existing reduced-motion philosophy from player-gestures.
Invariant

Under reduced motion (rememberReduceMotion(), inherited), the chrome fade is replaced by an instant show/hide and the Info sheet appears without its slide — animation disappears, behavior does not. Every action remains present, in the same slot, reachable by the same tap and the same a11y path. No action is only discoverable through motion; nothing about the action set changes when animation is off. This is the same rule player-gestures applies to gestures, extended to action presentation.

13

Frozen boundary — what this spec owns, what later specs own

The action framework is defined here once; every later player spec plugs into it. Nothing leaks across.
player-actions.html OWNS
  • The slot framework: primary cluster + leading/trailing secondary slots + the (empty) overflow slot.
  • The primary-action rule (transport is the sole primary; no hero CTA).
  • The secondary ceiling (Info + Share, both proposed) and every accept/reject verdict.
  • The overflow philosophy (frequency×reversibility; empty is valid).
  • The context-visibility rule (present-not-disabled) across states/shapes.
  • The destructive-action policy (defined here; placed in Library).
  • The a11y + motion contract for actions.
Later specs OWN (do not design here)
  • player-sharing.html — the Share flow: targets, SAF, and the DualShot which-side(s) picker. This spec only guarantees the Share slot.
  • player-info.html — the details sheet's full contents & layout (and any richer technical data, which would be new capability). This spec only defines the Info entry point + that a sheet exists.
  • player-editing.html — trim / edit UI. Not an action in this spec; if adopted it claims the reserved overflow slot, via its own HTML, re-approved first.
  • player-dualshot.html — angle swap and any DualShot-only control.
  • player-fit-fill.html — fit/fill display toggle.
  • MediaSession / media-key / lock-screen transport — a future backend decision (Media3 1.4.1 has none today). Out of scope everywhere until the backend exists.
Boundary rule

A later spec may populate a slot this framework defines (Share fills the trailing slot; editing may fill overflow), but it may not redefine the framework — the primary rule, the two-slot ceiling, the present-not-disabled visibility rule, and the destructive policy are frozen here. A change to any of those routes back through this HTML and is re-approved first, exactly like every other design change in the HTML-first workflow.
