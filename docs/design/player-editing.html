Rova · Player Redesign — "Playback to Standard"

Player Editing — the editing contract (a FUTURE capability)

Everything that happens after an editing action is chosen. Unlike Info, editing does not exist in Rova today and player-actions did not reserve an affordance for it — the fake Edit/Trim buttons were removed in PR-5 and the full editor is Parked NO-GO for v1.0. The whole app is copy/remux only: there is no frame-level transcode anywhere, and no replace-in-place semantics anywhere. So this spec designs the future editing experience honestly — every capability classified Existing (none) · Future · Rejected, nothing designed-around a gap. Its load-bearing decision: editing is non-destructive, save-as-a-new-copy only, which is exactly the grain of the codebase (all writes are new-file + atomic publish). It layers on the frozen player-core/states/gestures/actions/sharing/info, preserves ADR-0037/0038/0032/0025/0036, and is PROPOSED, gated behind a dedicated editing ADR — not implementable from this document alone.

Edit layer
00

Model & boundary — what editing is, and the wall it stands behind

The one-line model, the hard fact that no backend exists, the backbone it preserves, and the lines it may not cross.
The model in one line

Editing = a single, non-destructive trim that saves the kept range as a new video, leaving the original recording untouched. It is the smallest useful edit for a background recorder — "keep this stretch as a clip I can share" — not a timeline editor.

It is a copy, never a mutation: the source recording, its manifest, and its identity are read-only inputs; the output is a brand-new file published through the app's existing export path. No original is ever overwritten or deleted by editing.

The hard fact — nothing to build on today
Zero editing backend exists

No Transformer, no FFmpeg, no re-encode path. The only existing-file machinery is VideoMerger's sample-level remux. Replace-in-place semantics do not exist anywhere. The 17-tool editor is Parked NO-GO for v1.0.

This spec therefore designs a future feature and says so on every page: each capability is Existing / Future / Rejected, and the whole thing is gated behind a new editing ADR plus owner sign-off. It is not "ready to implement."

Frozen · what editing may NOT touch
  • Never mutate the archive. The source session's files, SessionManifest, segments, identity (ADR-0037), and vault state are read-only. Editing adds no schema field and rewrites nothing.
  • Never redesign the player, transport, timeline, layout, tokens, states, gestures, the action framework, sharing, or info — those are the six frozen specs. Editing only adds an output flow.
  • Never invent backend. No transcode, no new dependency, no MediaSession is assumed present; anything that needs one is classified Future/Rejected, not designed-around.
  • Never replace-in-place. "Save over the original" is rejected outright — the app has no such write and the original is provenance.

02

Editing philosophy — the archive is sacred, the copy is cheap

First principles for a background recorder that must never damage its own evidence, filtered through "no backend / non-destructive / calm."

Rova records unattended footage the user did not frame. That footage is often the point of keeping the app — a record of what happened while they were away. So the governing instinct is preservation: an edit must never put the original at risk. The competitive field splits cleanly on exactly this axis:

SourceWhat they doPrinciple Rova takes
Google PhotosNon-destructive trim; "Save copy" keeps the original, edits live in a copy.Adopt non-destructive, save-as-copy — the exact grain of Rova's new-file-only storage.
Apple PhotosTrim with revert-to-original — edits stored as a reversible sidecar over the original.Reject the mechanism revert-in-place needs an edit-history sidecar + in-place rewrite Rova has neither of; adopt only the non-destructive spirit, delivered as a copy.
DJI Mimo · BlackmagicMulti-track timelines, LUTs, speed ramps, export render.Reject a production editor is a different app; a background clip has no such craft intent.
CapCutFull effects/keyframe editor, heavy transcode.Reject far outside a hands-free recorder's scope; would demand the whole absent transcode stack.
Frozen philosophy
  • The original is never touched. Every edit produces a new file; the source recording, manifest, and identity are read-only. This is non-negotiable and matches the codebase (no in-place write exists).
  • One edit, done well: trim. Choose a stretch to keep, save it. Clip extraction is the same operation (a trim is extracting a sub-range). Everything heavier is Rejected (§03), not hidden in a menu.
  • The copy leaves the session graph. A trimmed clip is an ordinary saved video, not a new session — it carries no sessionId and needs no schema change (§04).
  • Honest about not existing yet. No page pretends the engine is present. The experience is designed so that when the ADR is approved, it drops onto real machinery — but it is Proposed until then.
03

Editing capabilities — what's in, what's out, each one classified

Every candidate edit tagged Existing · Future · Rejected against the repo fact-sweep. Exactly one is in scope; it is Future.
Critical · the starting point is empty

There is no editing capability in the app at all today — no trim, no cut, no extraction, no transcode. So nothing below is "Existing." The design question is not "which existing edits do we surface" but "which single new edit is worth building, and which do we reject before they metastasise into an editor." The answer: trim (save-as-copy), and nothing else.

Frozen decision · scope is one edit

The player's editing surface is trim-to-copy, and only that. Clip extraction is the same operation. Replace, merge, composite, and the effects suite are Rejected — each is either a new render Rova can't do or a mutation Rova must never do. Adding any of them is a new spec + new ADR, never an "in-code quick add." This keeps editing a single, safe, buildable increment rather than the door to an editor.

04

Artifact ownership — where the copy lives, and why nothing is rewritten

Save-as-copy vs replace, resolved by the codebase's own grain. The trimmed clip leaves the session model entirely.
Save as copy — required, and the only option
  • The output is a new standalone video, published through the export tiers' session-agnostic publish primitives — the MediaStore pending-insert + IS_PENDING flip on API 29+, the scoped-file + MediaScannerConnection scan on API 24–28 — the same low-level machinery the app already uses to put a file in the gallery. It does not route through the ExportPipeline/Tier1Exporter orchestrators, which are keyed on sessionId and whose commit points are manifest writes (see the frozen note below).
  • It is not a new session. It carries no sessionId, gets no SessionManifest, and is not in the recovery/session graph — so no schema field is added and no session invariant is touched.
  • The source is a read-only input. Editing opens the resolved artifact (ADR-0037 identity, transported) purely to read frames; it writes nowhere near the session directory.
Replace — rejected on the codebase's own terms
No in-place mutation
  • The app has no replace-in-place write anywhere — every write is new-file + atomic rename/publish; every delete is whole-artifact (ADR-0036).
  • The original is provenance — its manifest records what actually happened; overwriting it would falsify the record and could orphan identity/resume state.
  • A "replace" button would be a genuinely new, riskier mutation pattern for zero user benefit over "copy" — rejected.
Frozen decision · manifests are not updated by editing

Editing writes exactly one thing: a new published video, via a session-free publish that reuses the export tiers' publish primitives (pending-insert/flip · scoped-file + scan) — not the manifest-writing ExportPipeline orchestrators. The SessionManifest is never updated, extended, or rewritten by an edit — the trimmed clip is disowned from the session model on purpose, so editing needs no new schema field and no new persistence. (The export orchestrators commit through setExportPending/setExportFinalized = manifest writes, so editing bypasses them and calls the underlying publish directly — a small new session-free write path, expected since editing is new code behind its own ADR.) If a future product ever wants trimmed clips tracked inside the library/session graph (thumbnails, re-open, provenance link), that is a separate ADR adding a derived-artifact field — explicitly out of scope here, and not designed-around.

05

Editing execution — the engine that does not exist yet, classified honestly

How a trim is actually produced. Two real routes (both Future), one rejected. The UI above is engine-agnostic; the ADR picks the route.
RouteWhat it doesCost / dependencyStatus
A · Keyframe-aligned remuxSnap the trim to the nearest keyframes and sample-copy the kept range with MediaExtractor → MediaMuxer — an extension of the existing VideoMerger path. No decode, no re-encode.Cheapest — reuses machinery on the classpath. Trim points snap to GOP boundaries (not frame-exact). Still needs a new write path + UI that don't exist.Future (light)
B · Frame-accurate transcodeDecode + re-encode the boundary GOPs so the cut lands on the exact chosen frame, via media3-transformer.Adds the deliberately-excluded media3-transformer dependency + a real transcode (time, heat, battery on a background-recorder device).Future (heavy)
C · In-place edit of the source fileRewrite the original in place.—Rejected no in-place write exists; violates §04.
Recommended route for the ADR (not decided here)

Route A is the honest first increment: it reuses VideoMerger's sample-copy, adds no dependency, and is safe. Its one visible compromise — trim points snap to keyframes, so the cut can be off by up to a GOP — is disclosed in the UI ("the start/end may shift slightly to the nearest full frame") rather than hidden. Route B (frame-exact) is a later, separate ADR only if users demand precision. The specimen above is drawn to be true under either route; the sheet doesn't change, only the engine behind "Save copy" does.

Execution flow (when built)
  • Preflight storage exactly like export (enough free space for the copy) before starting — reuse the export storage check.
  • Run off the main thread; the player is paused, the sheet shows determinate progress (the app already has a merge/export progress idiom).
  • Publish atomically — write to a temp/pending target, then rename/flip-pending exactly as the export tiers do; the copy only becomes visible when whole.
  • Confirm + offer share on success (hand off to player-sharing), leaving the player exactly where it was.
Playback while editing
  • Choosing edit pauses playback; the trim sheet is a modal over the paused frame. No autoplay.
  • ADR-0038 lifecycle is unchanged — same leased player, no new lifecycle owner; editing does not touch PlayerEngine.
  • Scrubbing the trim handles uses the existing seek (player-gestures) to preview boundaries — no new transport.
  • Cancel or dismiss returns to normal playback with the playhead intact.
06

Failure handling — the original is the safety net

Because editing only ever writes a new file, every failure mode is non-destructive by construction. Reuse the export failure vocabulary.
FailureWhat the user seesWhat is true of the original
Not enough storageCaught at preflight before work starts: "Not enough space to save the copy." No partial output.Untouched — same as export's storage guard.
Trim engine error (mux/transcode fails mid-run)"Couldn't save the trimmed copy." Retry offered. The pending/temp target is cleaned up.Untouched — the source was only read.
Process killed mid-editNothing to recover — the edit was never committed; the pending file (if any) is orphaned temp, swept like export temp.Untouched — no session state was written, so recovery has nothing to reconcile.
Publish (rename/flip-pending) failsTreated as a failed save; the incomplete public row/temp is removed, same as a failed export publish.Untouched
Empty / invalid rangeSave is disabled until the range is valid (min length enforced). No failure to handle.Untouched
Frozen decision · failure is always safe

Because editing never mutates the source and only ever publishes a new file, the worst case is "the copy didn't get made" — the original is always intact and the session graph is never in a half-edited state. There is no edit-specific recovery pipeline: an interrupted edit leaves at most an orphan temp/pending file, swept by the same cleanup the export pipeline already runs. Editing adds no new terminal state, no new manifest field, and nothing for the cold-launch recovery scan to classify.

07

DualShot editing — one angle, the one on screen, never composited

Two independent files, one sessionId. Editing operates on the reviewed side only, reads existing per-side data, reconstructs no identity.

A DualShot session is two finalized files (portrait + landscape) sharing one sessionId, settling independently. The player is reviewing one side (its transported PlaybackIdentity.side, ADR-0037). Editing follows the same rule as everything else in the player: it acts on the artifact currently on screen.

QuestionAnswerWhy
Which side does trim edit?The reviewed side only (the open PlaybackIdentity.side).Show no ambiguity, no identity minted — edits what's on screen.
Can you trim both at once?No — one angle at a time; the other side is untouched.Simple each side is a separate file; switch angle (player-dualshot) then trim the other.
Composite / side-by-side / picture-in-picture output?No.Reject a two-source render — needs the absent transcode + compositor (§03).
Does trimming one side change the other, or the session?No — the output is a disowned copy; both source files + the manifest are untouched.Safe §04 holds identically for DualShot.
What if the reviewed side isn't finalized?Edit is unavailable for that side; the sheet explains "this angle isn't ready to edit."Honest sides settle independently; no fabricated source.
Frozen decision

DualShot editing is single-side, on the reviewed angle, producing a disowned copy of that one file — exactly like the single case. It never composites, swaps, or combines the two angles (that would be a new render, Rejected §03) and it reconstructs no identity (ADR-0037: transported, not reconstructed). To trim the other angle, the user switches to it (a player-dualshot concern) and trims it there. The session's two files and manifest are read-only throughout.

Frozen accessibility floor

The trim range must be settable without dragging — start/end are keyboard/switch-adjustable sliders with spoken time + resulting length, 48dp targets, and an enforced minimum so they never collapse. The sheet is a focus-trapped dialog returning focus to the trigger on close; every row is a single labelled node; the "original is untouched" reassurance is spoken. Progress and both outcomes are announced. A control that can only be operated by gesture is not shipped — it is reworked until it isn't.

09

Motion — reuse the sheet and progress vocabulary, add nothing

Editing introduces no new motion language; it is the same pinned sheet + the same progress idiom the app already moves.
Motion (inherited only)
  • Sheet enter/exit on inherited timings (--t-container / --t-exit, --ease-std) + swipe-down-dismiss — identical to sharing/info.
  • Handle drag is direct manipulation (no animation); the dimmed out-of-range mask updates live with the handle.
  • Saving uses the existing determinate merge/export progress motion — nothing bespoke.
  • No new motion token. Anything that seems to need one is a player-core change first, gated behind the editing ADR.
Why so little

The only genuinely new interaction is dragging a range, which should feel like direct manipulation, not choreography. Everything around it (sheet, progress, confirm) is already defined elsewhere; editing reuses it so the flow feels like one continuous player, not a separate editor.

10

Reduced motion — animation disappears, the edit does not

The inherited invariant, extended to editing.
  • Under rememberReduceMotion(), the sheet appears without its slide; dismiss is instant.
  • The trim range, both handles, result rows, and Save/Cancel remain in the same place, reachable by the same tap, drag, arrow-step, and a11y path.
  • Progress becomes a static determinate indicator (no animated sweep) — the percentage/step still updates and is still announced.
  • No part of editing is discoverable only through motion — the range and its outcome are legible from static state alone. Same rule as gestures/actions/sharing/info: animation changes, behavior and content do not.
11

Frozen boundary — what this spec owns, what stays elsewhere, what needs an ADR

Editing owns the trim-to-copy experience. Everything structural is Future capability behind a dedicated ADR.
player-editing.html OWNS
  • The trim-to-copy experience: the range-selection sheet, its grouping and order, the single Save copy commit.
  • The scope decision (§03) — trim/extraction in; replace/merge/composite/effects out.
  • The artifact model (§04) — non-destructive, disowned copy via the existing export publish, no manifest change.
  • The execution classification (§05) — Route A/B Future, in-place Rejected — and the paused-playback, ADR-0038-preserving flow.
  • The failure model (§06, safe-by-construction), DualShot single-side rule (§07), and the editing a11y + motion contract.
NOT owned here (do not design)
  • The edit affordance itself — whether/where an edit action appears — is player-actions.html. It does not exist today; this spec only defines what it would open.
  • The trim engine (keyframe-remux vs transformer) — a new implementation ADR, not chosen in this HTML.
  • Any tracked/derived-clip schema — if trimmed clips are ever put back into the session/library graph, that is a separate schema ADR.
  • player-sharing (the post-save share), player-info, player-dualshot (angle switching), transport, gestures, states.
  • The full editor (speed/filters/text/transitions/merge) — Parked NO-GO; a different product, not this spec.
Boundary rule

This spec defines the experience of a non-destructive trim so that, if approved, it drops cleanly onto Rova's real grain (new-file publish, no in-place mutation, no schema change). It consumes the resolved artifact (ADR-0037), the leased player (ADR-0038), and the export tiers' session-free publish primitives (not the manifest-writing orchestrators) — and redefines none of them. The editing contract — trim-to-copy only · original never mutated · disowned output via export · single-side DualShot · engine classified Future · replace Rejected — is frozen here; a change routes back through this HTML and is re-approved first. Editing remains PROPOSED and is not buildable from this document alone: it requires a dedicated editing ADR (engine + affordance) and owner sign-off before any Compose work.
