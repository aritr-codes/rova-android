Rova · Player Redesign — "Playback to Standard"

Player Accessibility — the contract that makes eight specs one system

This is the final Player specification and it is a contract, not a feature set. Every earlier Player spec already carries its own accessibility section; left alone, that is eight independent a11y sections that can drift, double-speak, or disagree. This document states the one accessibility system they are all instances of — the semantic tree, the single focus order, the live-region ownership rule that stops two nodes talking over each other, and the large-text · scaling · contrast · caption · motion · transparency policy — grounded in the real composables (PlayerScreen.kt, SegmentedTimeline.kt) and the two real seams (rememberReduceMotion, rememberReduceTransparency). It adds no accessibility feature and redesigns no surface: it names what is shipped, what is a grounded gap, and what each later spec must satisfy. It preserves ADR-0037/0038/0032/0025 and requires no backend, no schema change, no MediaSession, no Media3 change. WCAG 2.2 AA (ADR-0020) is the floor, not a pass.

A11y model
00

Model & boundary — one system, stated once

What this contract is, what it is emphatically not, and the backbone it may not touch.
The model in one line

Player accessibility = every Player surface exposes the same semantic contract: a single labelled control per affordance, one predictable focus order, one live-region owner per concern, and one motion/transparency/contrast/scaling policy — so a TalkBack, keyboard, D-pad, or Switch-Access user experiences one player, not eight screens that each did accessibility their own way.

It is the invariant; each surface spec's a11y section is an instance of it. Where they already agree (they do — all eight require non-gesture parity, 48dp, reduce-motion, assertive failures), this document is their union, not a change.

What this document is NOT
Not a feature spec

It adds no accessibility feature, redesigns no layout, playback, gesture, action, sharing, info, editing, DualShot, transport, or state. Those are authoritative and frozen. This spec only unifies and grounds their accessibility.

Every requirement here is either Shipped (cited in source), a grounded gap (an absence finding the owning spec must close in its transcription), or a Future obligation attached to a surface that is itself Proposed (sheets, DualShot switch). Nothing is invented.

Frozen · the walls this contract stands behind
  • No backend, no schema, no MediaSession, no Media3 change. Accessibility here is semantics, focus, and announcements over the existing composables. Media3 stays 1.4.1; there is no media-key / lock-screen transport and this contract does not assume one (that would be a separate backend ADR — player-actions §13).
  • Backbone read-only. ADR-0037 (identity resolver), ADR-0038 (leased player), ADR-0032 (wall-clock), ADR-0025 (vault, FLAG_SECURE) are untouched. A screen reader never changes what identity is resolved or how the lease behaves.
  • The owning spec owns its surface. If satisfying this contract needs a visual or interaction change to a surface, that change is made in that surface's HTML first and re-approved — never invented here. This document sets the semantic floor; it does not move a pixel.
  • WCAG 2.2 AA is the floor (ADR-0020). Not a target to reach later — the baseline every Player surface is measured against.
01

Accessibility philosophy — the player is small on purpose, and that is the asset

First principles for making an over-media, near-monochrome, gesture-light player fully operable by everyone.

Rova's player is deliberately minimal: one primary control, a few demoted peers, a spine, no hero CTA, no clutter. For accessibility that minimalism is not a risk to paper over — it is the strongest asset. Fewer controls means fewer things to label, fewer focus stops, fewer states to announce, and no maze to traverse. The whole philosophy is: keep the surface small, make every element on it a real, labelled, non-gesture control, and let the reader hear the same simple structure the sighted user sees.

Accessibility is designed into the smallness, not bolted onto it. The player earns its reachability by having few, honest, labelled controls — so the contract's job is mostly to guarantee that smallness stays fully operable across TalkBack, keyboard, D-pad, Switch Access, large text, and reduced motion/transparency, and that the eight surfaces never diverge in how they do it.

03

Focus order — one linear path, sheets trap and return

The traversal order for TalkBack swipe, D-pad, keyboard Tab, and Switch scan. It is the reading order; an open sheet is a modal that traps focus and hands it back.

Today there is no explicit traversal ordering in the Player — no traversalIndex, isTraversalGroup, or FocusRequester anywhere — so order is implicit composition order, which happens to read correctly. This contract makes that order explicit and normative so no future edit reshuffles it, and defines how modals interrupt it.

The base order (chrome visible)
  • 1 · Back — top-left, first stop (escape is always one swipe away).
  • · Read-only text stops — the InfoRow position + clip-counter (PlayerScreen.kt:544,553) and the wall-clock readout (:623, the §02 node) are non-interactive Text stops that fall in reading order between Back and the timeline — swiped over, not activated. They are stops, not focusable controls, so they carry no number.
  • 2 · Timeline — the spine; adjustable, before the transport (matches actions §10).
  • 3 · −10s → 4 · Play/Pause → 5 · +10s — the transport cluster, left-to-right.
  • 6 · Speed — the demoted trailing peer.
  • (7·8 · Info / Share) — appended in the trailing slot only when those Proposed actions exist (actions §10 order: … → speed → Info → Share).
Modals (sheets) — trap & return
  • Opening a sheet (Info / Share side-picker / Trim / any Proposed panel) traps focus inside it; nothing behind is reachable.
  • Reading order inside the sheet is top-to-bottom, primary action last; a labelled close/handle is reachable.
  • Back/Esc/dismiss closes the sheet and returns focus to the control that opened it — never dumps focus to the top of the screen.
  • This is owed as FocusRequester wiring (none exists in Player today) and is a per-sheet obligation on the owning Proposed spec.
Angle switch (DualShot) — a focus subtlety

On a same-moment angle switch (player-dualshot §07), focus stays on the switch control across the re-lease so the user can immediately reverse — it does not reset to the top. An unavailable side is a state node, skipped in focus order, not a dead button. Both are that spec's obligations; this contract only fixes that a re-lease must not lose focus.

Frozen decision · the order is normative

Back → Timeline → −10 → Play/Pause → +10 → Speed → (Info → Share when present). It is the reading order and it is fixed here — a transcription may not reorder it, and a new control claims a defined slot (trailing, after speed) rather than wedging into the transport. Sheets trap focus and return it to their trigger. Focus survives a DualShot re-lease. No control is ever unreachable by swipe, Tab, D-pad, or scan.

Frozen decision · one fact, one voice

Each state is announced exactly once, by exactly one mechanism: play/pause by its swapped label, position by the timeline's stateDescription, segment changes by the one live region, failures by the state announcer. A transcription must not add a second node that re-announces a fact already carried — the most common a11y regression on a rich player is double-speak, and this contract forbids it structurally (§05).

05

Live regions & announcements — one owner per concern, one loud voice at most

The systemic rule no single surface spec could make: which node speaks, how loudly, and what must stay silent.

Multiple specs add announcements — the timeline speaks segment changes (shipped), player-states adds Loading/Ready/Unavailable, resume speaks a cue. Uncoordinated, they collide: a segment jump during a load could fire two polite regions at once, or an assertive failure could be buried under a polite tick. This contract assigns a single owner per concern and one politeness ladder.

ConcernOwner nodePolitenessRule
Segment index changeTimeline live region (SegmentedTimeline.kt:250)PoliteShipped throttled to index changes only — never per-frame.
Loading (escalated)State announcer (states §08)PoliteOwed only after the grace delay, so a fast load stays silent.
ReadyState announcerPoliteOwed the clip title / "Ready"; no interruption.
Unavailable (failure)State announcerAssertiveOwed a failure must never be silent — the ONE place assertive is used.
Resume cuePlay control label ("Resuming at m:ss")Polite (on the label)Owed spoken via the control, not a separate region.
Angle switched (DualShot)The switch control (announced on change)PoliteFuture "Switched to Landscape, same moment" (dualshot §07).
Scrub position / elapsed time— none —SilentBy design the reader is driving the seek; echoing every position is chatter. The stateDescription is read on focus, not announced live.
Frozen decision · the announcement ladder
  • Assertive is reserved for failure (Unavailable) and nothing else. At most one assertive announcement is live at a time.
  • One owner per concern. Segment changes belong to the timeline node; stream state belongs to the single state announcer; playback state belongs to the play/pause label. No concern is announced by two nodes.
  • Silence is a valid choice. Scrub position, elapsed time, and per-frame progress are not live — they are readable on focus. A player that narrates every second is unusable.
  • Throttle. Any live region fires on a meaningful change (segment index, state transition), never on a continuous value.

Keyboard · D-pad · Switch Access — the non-gesture guarantee

Every player function is reachable and operable without a touch gesture. This is the hard promise every surface spec already makes; here it is the system law.

The player is gesture-light, and every gesture is an accelerator over a visible control — so the non-gesture story is mostly "operate the visible controls." This contract makes explicit that the same linear focus order (§03) is the keyboard Tab order, the D-pad traversal, and the Switch-Access scan order, and that the one genuinely gesture-only accelerator (double-tap ±10s) has a visible equivalent.

Grounded note · targets & the gate's blind spot

Every control is ≥48dp: RovaTokens.minHitTarget = 48.dp on the timeline row (SegmentedTimeline.kt:126), inline .size(48.dp) on play/pause + speed (:679,:708), and the Material IconButton 48dp default on Back/±10s. The static gate ruleA11yTargetSizeToken checks a curated token set at a 24dp floor and no Player size is in that set — so the Player's 48dp targets are correct but not gate-guarded. The contract's floor is 48dp (above the 24dp SC 2.5.8 minimum); protecting it is a transcription concern (a pure test or adding the tokens to the gate set), not a new gate demanded here.

Frozen decision · no gesture-only function

Every player function has a focusable, labelled, ≥48dp control operable by keyboard, D-pad, and Switch Access, in the §03 order. Gestures are shortcuts layered on top; the single gesture-only accelerator (double-tap seek) is mirrored by the ±10s buttons. A control that could only be driven by a gesture is not shipped — it is reworked until it isn't (the rule gestures/actions/sharing/dualshot all state; unified here). Media-key / lock-screen transport is explicitly out (no MediaSession, Media3 1.4.1) — the guarantee is over the in-app controls.

07

Large text & display scaling — text grows, targets don't, nothing clips

A dimension no earlier spec covered in depth. Font scale and display size must not truncate chrome or shrink a control below 48dp.

Android users can raise font scale (up to 2.0×) and display size independently. The over-media chrome is dense — a two-line title, small metadata, a speed multiplier — so it is exactly where large text can clip or overlap. The contract: text is in sp and reflows; interactive targets are in dp and do not scale with font; no fixed-height text container clips its content.

The rules
  • All text in sp — it honours the user's font scale (Player text already uses .sp).
  • Targets in dp — the 48dp hit areas are independent of font scale; a bigger label must not inflate (or shrink) the touch target.
  • No clipping. The two-line title wraps or ellipsizes gracefully rather than being cut by a fixed height; the timeline row's heightIn(min=48dp) is a minimum, free to grow.
  • Chrome bands must not clip. The bottom band already hugs its content height (core §08). The top band is a fixed .height(96.dp) today (PlayerScreen.kt:408) — exactly where a wrapped two-line title can clip under large text; converting it to a min-height floor is owed large-text work (see gap panel), not current behavior.
  • Display size (density) scaling is honoured by using dp/sp throughout — no px, no hard-coded pixel geometry.
Grounded gaps & test obligation
  • Player text sizes are inline .sp literals (PlayerScreen.kt, SegmentedTimeline.kt) — correct unit, but there is no test asserting the chrome survives fontScale 2.0.
  • The two-line title is ungrouped and single-line-ellipsized today; grouping it (§02) and confirming it wraps under large text is the transcription obligation.
  • The top band's fixed .height(96.dp) (:408) should become a min-height so a wrapped title grows the band instead of clipping — owed large-text work.
  • Verify on device at fontScale 2.0 + display "largest" — part of the standing device-verification gate, not a claim provable in HTML.
  • No nonScaledSp / fixed-dp text anywhere — the player must never opt text out of scaling.
Frozen decision · scaling is a first-class state

The player is legible and fully operable at the maximum font scale and display size. Text scales and reflows (sp, wrap, no fixed-height clip); touch targets stay a constant 48dp (dp); no control drops below the floor and no label is truncated into meaninglessness. This is verified on device at max scale as part of the release gate — it is not a "best effort," it is the floor.

08

Motion / reduced motion — one gate, nothing means anything by moving

The single motion seam, unified across every surface. Reduced motion strips animation and removes zero information.

There is exactly one motion gate in the app — rememberReduceMotion() (ReducedMotion.kt:47) — and the Player consumes it in both its animated places. Every surface spec reduces motion the same way; this contract states the invariant once.

What animates (inherited only)
  • Chrome fade (both bands) — AnimatedVisibility gated at PlayerScreen.kt:402-403,480-481.
  • Timeline playhead glide — RovaMotion.containerSpring(), gated at SegmentedTimeline.kt:197.
  • Control press, poster→first-frame cross, and any future sheet slide — all on the inherited ladder (--t-micro/-small/-container/-exit, --ease-std).
  • No new motion token is introduced by accessibility.
Under reduced motion
  • Chrome appears/disappears instantly (EnterTransition.None) — the reveal still happens.
  • Timeline fill snaps (also while scrubbing) — the seek still lands.
  • Poster→frame is a hard cut; spinner (states) becomes a static ring; any sheet appears without its slide.
  • Behavior and content are identical — only animation is removed. Every affordance in §02–§07 stays exactly where it is.
Gate · infinite animations must self-gate

Any looping animation (a spinner, a pulse) is caught by ruleA11yAnimationGated — it must co-locate rememberReduceMotion()/ReducedMotion.isReduced or it fails the build (SC 2.3.3/2.2.2). So a future Player spinner (states' escalated Loading) cannot ship un-gated. Haptics are not motion and are gated separately by system haptics settings.

Frozen decision

One gate, one rule: nothing on the player carries meaning by motion. Disabling animation never removes information, never hides a control, never changes behavior — it only removes the animation. This holds for every current and future Player surface; a surface that needed motion to convey meaning would be a design defect routed back through its own HTML.

09

Transparency & contrast — proven over footage, opaque when asked

Reduced transparency reuses the shipped seam; contrast is proven by one worst-case composite pin, never eyeballed.
Reduced transparency
  • The seam exists: rememberReduceTransparency() (ReducedTransparency.kt:58) + RovaTrustTokens.pinContainerAlphaFor — the Trust banner/snooze chip already consume it (WarningTopBanner.kt:105, WarningSnoozeChip.kt:113).
  • Grounded gap: the Player does not consume it — its scrims are hard-coded Color.Black.copy(alpha) (PlayerScreen.kt:412-413,489-490), and its pinned glyph pills use a fixed --glyph-fill.
  • Contract: under OS reduce-transparency, the Player's pinned pills (the transport-glyph backing) go opaque via the same pinContainerAlphaFor path the Trust surfaces use — the transcription obligation core §04 already implies. The gradient bands are a legibility scrim, not a decorative blur; they stay (their job is contrast, and removing them would hurt a low-vision user).
Contrast guarantee
  • Over-media ink sits over arbitrary, worst-case-bright footage — it can't be tuned per-literal.
  • The one obligation (core §07): a scrim-composite worst-case contrast pin — a pure JVM test asserting --media-ink-dim (.48) and --media-ink-body (.55) clear AA over the band-over-bright-video composite, ×palettes, modelled on TokenContrastTest.mediaProgress and extending OverlayContrastTest's dark-ref model to bright footage.
  • Grounded gap (narrowed): OverlayContrastTest.kt already proves the player subtitle (.72, :57-70) and play-button fill (.35, :72-81) — but only over a fixed dark-scene ref (#0E1216), never worst-case-bright video; the remaining .45/.55/.88 alphas are untested. So the gap is no bright-footage / scrim-composite proof, not "no test." Neither TokenContrastTest nor TrustContrastSweepTest covers the Player. Do not reuse the opaque-surface Trust sweep as the proof (it models opaque backings, not a translucent scrim over video).
  • The focus ring is white (--focus-ring), legible on any frame — not the accent.
Frozen decision

Contrast is proven, transparency is honoured, both via existing machinery. One scrim-composite pin is the Player's contrast guarantee (no per-literal tuning, no borrowing the opaque Trust sweep). Reduce-transparency routes the Player's pinned pills through the same pinContainerAlphaFor seam the Trust surfaces already use; the legibility bands stay because they serve contrast, not decoration. No new token, no new test family — the pin joins the existing sweep family (no new gate, per the ADR-0036/0038 precedent).

10

Captions / transcript — none exist, and that is stated, not designed-around

The honest policy for a background recorder whose footage carries ambient audio but no authored caption track.
The grounded fact

Rova has zero caption/subtitle machinery: no SubtitleView, no SubtitleConfiguration, no Media3 text track, no closed-caption path (only media3-exoplayer + media3-ui on the classpath). Recordings carry an ambient audio track (the live-capture AudioRecord) but no authored captions and there is no speech-recognition/transcript backend. Nobody wrote a caption track, so the player has none to render.

QuestionAnswerWhy
Does the player render captions?No — there is no caption track to render.Honest unattended footage has no authored subtitles; inventing an empty CC toggle would be a dead control.
Auto-generate a transcript (ASR)?No.Future needs an ASR backend the app doesn't have and this contract can't assume — a separate ADR + feature, not designed-around here.
Is audio-conveyed info lost to a deaf user?Recordings are ambient/hands-free video — their meaning is visual, not narration. The important state (playback, position, failures) is all conveyed visually and via TalkBack, not through audio.Adequate no information is audio-only; the app's own cues (beeps) are recording-time, not playback content.
Burned-in captions on export?No.Reject a render/transcode the copy/remux pipeline can't do (player-editing §00) and a product decision, not an accessibility default.
Frozen decision · caption policy

The Player renders no captions because Rova's recordings contain no caption track, and it exposes no CC control rather than a dead one. No transcript/ASR is assumed or designed-around. Because playback state and failures are conveyed visually + via TalkBack (never audio-only), a deaf or hard-of-hearing user loses no player information. If captions or transcripts are ever wanted, that is a new feature behind its own ADR (a caption/ASR backend), explicitly out of scope here — stated plainly, not worked around.

Frozen boundary — the contract this document owns, the surfaces it does not touch

This is the accessibility invariant. Each surface spec's a11y section is an instance of it; none is reopened here.
player-accessibility.html OWNS
  • The accessibility floor (§00–§01): WCAG 2.2 AA by default, the non-gesture guarantee, the "one fact, one voice" rule.
  • The unified accessibility tree (§02) and the normative focus order + sheet trap/return (§03).
  • The screen-reader label contract (§04) and the live-region ownership + politeness ladder (§05).
  • The keyboard/D-pad/Switch parity law (§06), large-text/scaling policy (§07), motion unification (§08), transparency/contrast guarantee (§09), and caption/transcript policy (§10).
NOT owned here (do not redesign)
  • Any surface's layout, controls, or interaction — player-core/states/gestures/actions/sharing/info/editing/dualshot. Each owns its own a11y section as an instance of this contract; this document never moves their pixels.
  • The specific announcements for Loading/Ready/Unavailable/resume — authored by player-states §08 (this contract only sets the ownership + politeness rule they follow).
  • The DualShot angle-switch semantics — player-dualshot §07 (this contract only fixes focus-survives-re-lease and one-owner announcing).
  • Sheet contents (Info/Share/Trim) — their owning Proposed specs (this contract only requires focus-trap + return + one-node-per-row).
  • MediaSession / media-key / ASR / caption backend — future backend ADRs, out of scope everywhere.
Boundary rule

This document freezes the one accessibility system the whole Player conforms to — AA floor · one node per affordance · normative focus order · one live-region owner per concern · assertive only for failure · non-gesture parity across TalkBack/keyboard/D-pad/Switch · text scales while 48dp targets don't · one motion gate · contrast proven by a scrim-composite pin · reduce-transparency via the shipped seam · no captions because none exist. It consumes the resolved identity (ADR-0037), the leased player (ADR-0038), the wall-clock playhead (ADR-0032), the vault (ADR-0025), and the two existing seams — and redefines none of them. It requires no backend, no schema change, no MediaSession, no Media3 change. Everything Shipped is cited in source; every gap is grounded; every Future obligation rides a surface that is itself Proposed. A change to this contract routes back through this HTML and is re-approved first. The Shipped guarantees hold today; the owed items (title grouping, state announcer, focus-trap wiring, the scrim-composite contrast pin, the reduce-transparency wiring, the large-text device check) land with their owning surface's transcription — this contract names them; it does not implement them.
