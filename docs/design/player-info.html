Rova · Player Redesign — "Playback to Standard"

Player Info — the recording-information contract

Everything that happens after Info is pressed. player-actions.html defined Info as the second permanent action and deferred its contents here. This spec layers on the frozen player-core / player-states / player-gestures / player-actions / player-sharing and inherits every token, layout, state, gesture, action, and the sheet idiom — it never redraws them. It decides what information is shown, and where each field comes from. Grounded in current source (the player already holds durations/clips/wall-starts; the manifest holds the rest; fps/codec/bitrate are never stored and are rejected, not designed-around) + competitive metadata research — not memory. Preserves ADR-0037/0038/0032/0025, and adds no backend capability.

Info layer
00

Model & boundary — what Info is, and what it must not touch

The one-line model, the backbone it preserves, and the redesign lines it may not cross.
The model in one line

Info = a calm, read-only account of what this recording is and how it came to be, drawn from data the app already has. It is a review aid — provenance and context first, technical trivia last (or never) — not a camera-app spec sheet.

It is a read, not a probe: everything shown is already in memory (the player's state) or one cheap disk read away (the manifest). Info performs no file decode by default, invents no metadata, and never writes anything.

What this spec must NOT do
  • Not move or restyle the Info affordance, the Share slot, transport, timeline, layout, tokens, states, or gestures — those are the five frozen specs.
  • Not invent backend: no new persistence, no new schema field, no metadata the app doesn't already produce, no MediaSession.
  • Not design sharing (player-sharing), editing/trim, or DualShot playback controls — Info only describes.
  • Not change the resolver, export tiers, the vault model, or identity transport.
Frozen · Info adds NO backend capability

Every field routes something that already exists: the player's PlayerUiState.Ready (durations, clip count, wall-starts, session start), or a cheap re-read of manifest.json via sessionStore.loadManifest(sessionId) (mode, orientation policy, size, terminal/stop reason, vault state, per-side fields). The player gains exactly one new capability boundary — an Info entry point that re-loads the manifest and renders it read-only — and it remains PROPOSED, gated on owner sign-off (the same gate player-actions put on the Info action).

02

Information philosophy — provenance over spec-sheet

First principles for a background-recorder's review player, filtered through "no backend / calm / truthful."

Rova records unattended, in the background, in periodic clips. The user did not frame a shot — they set it running and came back. So the first question Info must answer is not "what codec" but "is this recording whole, and what happened while I was away?" Competitive galleries and pro-camera apps sit at two poles:

SourceWhat they showPrinciple Rova takes
Google Photos / Apple PhotosDate, time, size, resolution, a map, device — light, human, one screen.Adopt lead with human facts (when, how long, how big); keep it to one calm sheet.
Samsung GallerySame, plus an expandable "Details" with path/resolution.Adopt progressive: the everyday facts up top, the technical few below — never a wall.
Blackmagic Camera / DJI MimoCodec, bitrate, fps, ISO, shutter, white balance, LUT — a production sheet.Reject Rova is not a cine tool; it doesn't capture or store these, and a background clip has no framing intent to report.
Frozen philosophy
  • Provenance is the headline. The first thing in the sheet is what happened to this recording — completed, stopped by you, stopped by the system, force-stopped, or clips-kept-after-interruption. That is the fact a background recorder's user most needs.
  • Human facts before technical facts. When, how long, how many clips, how big — then mode/quality/interval. Codec-tier trivia is rejected outright (§04), not merely buried.
  • Only what the app truthfully knows. Every field is a real value from Ready or the manifest. Nothing is inferred, estimated, or fabricated to fill a row; unknown is shown as unknown (§03).
  • Read-only and calm. No controls in the sheet, no probing the file, no writes. Info describes; other specs act.
03

Field inventory — every field, and exactly where it comes from

Each field labelled ALREADY-LOADED · CHEAP-READ · EXPENSIVE-READ · REJECTED, per the repo fact-sweep. Nothing here needs new backend.
Critical · the player threw the manifest away

PlayerViewModel keeps only PlayerUiState.Ready + sessionId after resolving; it does not retain the full SessionManifest. So Info renders ALREADY-LOADED fields from Ready immediately, and re-loads the manifest once (sessionStore.loadManifest(sessionId) — cheap disk JSON, no decode) for the CHEAP-READ fields. That single re-read is the entire "backend" Info adds. No field below requires a schema change.

Frozen decision · availability drives inclusion

A field appears in Info iff it is ALREADY-LOADED or CHEAP-READ. EXPENSIVE-READ (actual resolution) is deferred behind a possible future opt-in, never run just to fill the sheet. DOES-NOT-EXIST fields (fps/codec/bitrate/location/device) are rejected — the honest answer to "why isn't fps here" is "the app never captured it," and Info does not fabricate. Any field that would need new capture or storage is a future ADR, not a row added here.

04

Technical metadata — what's honest to show, what's rejected

The one place a spec-sheet instinct meets the repository. The repository wins: most of it does not exist.
Quality — shown, but honestly

config.resolution is the requested picker label ("SD/HD/FHD/4K"). The manifest's own KDoc is explicit: it is not updated to reflect what the device actually delivered. So Info shows it labelled "Quality (requested)" — truthful about being an intent, not a measurement.

The actual delivered resolution exists only via a MediaMetadataRetriever decode (VideoMetadataUtils), which today runs only for Library/Vault rows, never in the player. Surfacing it here would mean a file decode on every Info open — rejected as the default. It is deferred to a possible future "verify actual quality" opt-in, decoded on demand and cached, never eagerly.

fps · codec · bitrate — rejected
Not stored, not shown
  • fps — never extracted, never stored, anywhere in the app.
  • codec — only a record-time VideoCodec{H264} enum exists in the dual-recorder types; it is not written to the manifest and does not describe the merged output.
  • bitrate — never extracted or stored.

Showing any of these means either a per-file decode (cost, jank) or a fabricated constant (a lie). Both are rejected. A background recorder's user gains nothing from a codec string; the calm sheet is worth more than the completeness.

Frozen decision · reject, don't design-around

Info shows requested quality (labelled as such) and no other technical field. fps, codec, and bitrate are rejected — the app does not have them, and Info never invents data. Actual delivered resolution is deferred to a future decode-on-demand opt-in with its own decision; it is not part of v1 Info. This keeps Info truthful, instant, and jank-free.

05

Recording provenance — the headline the sheet leads with

A background clip's most important fact is what happened to it. Mapped from the real terminal enums.

The provenance banner is a plain-language reading of manifest.terminated (and, for stops, stopReason). These are the actual enum values — no invented states:

terminatedBanner (plain language)Detail line (from stopReason where relevant)
COMPLETEDCompleted normally"Ran to the end of its loop." Neutral/positive tone.
USER_STOPPEDStopped by you"You ended this recording early." (stopReason.USER)
KILLED_BY_SYSTEMStopped by the systemReason-specific: low storage / thermal / permission revoked / schedule window ended — mapped from stopReason.
KILLED_FORCE_STOPForce-stopped"The app was force-stopped during recording."
MULTI_SEGMENT_KEPTClips kept after an interruption"Clips recorded before the stop were kept and are playable." (§07)
null (still live — not reachable in the player)—The player only opens finalized/kept recordings; a live session is never in Info.

StopReason → detail (for system/early stops): LOW_STORAGE "Storage ran low" · THERMAL "The device got too warm" · PERMISSION_REVOKED "A permission was turned off" · INIT_FAILED "The camera couldn't start" · SCHEDULE_WINDOW "The recording window ended" · USER "You ended it" · NONE no detail line.

Frozen decision
  • Provenance leads the sheet — one banner, plain language, tone matched to the outcome (neutral for completed, gently informative for interrupted, never alarming).
  • Every banner maps to a real enum (Terminated × StopReason). No state is invented; an unmapped combination falls back to a neutral "Recording finished."
  • Vault is not an interruption — a vaulted session is a normal, playable recording rendered under FLAG_SECURE. Its provenance banner is its real terminated outcome like any other. Info is read-only and leaks no file, so it opens on vault like any resolved artifact; whether the Info action is present at all is player-actions' presence rule (which today says actions ride with the resolved artifact), not this spec's to override.
06

DualShot information — two angles, one recording, no invented identity

The player opened one side. Info describes both, reading existing per-side manifest fields — it reconstructs nothing.

A DualShot session is two finalized files — portrait and landscape — sharing one sessionId, with segments interleaved in one list discriminated by SegmentRecord.side, and parallel per-side manifest fields (portrait*/landscape*). The player is reviewing one side (its transported PlaybackIdentity.side). Info adds an Angles group so the whole recording is legible without leaving the reviewed side.

What Info shows for DualShotSourceDecision

Frozen decision

DualShot Info shows both angles as two rows (size per side + total, clips, length) with the reviewed side marked (reviewing). All of it reads existing data — side-filtered Ready for the open side, the already-re-read manifest's per-side fields for the other — so it reconstructs no identity (ADR-0037: transported, not reconstructed). If one side has no finalized artifact, its row shows the honest state (e.g. "not finalized") rather than a fabricated size. Info never composites, swaps, or plays the other angle — those belong to player-dualshot.

07

Recovery & unavailable information — explain the gap, don't hide it

Interrupted recordings, kept-raw segments, and any field the app truthfully doesn't have.
Interrupted / kept-raw sessions
  • The banner names the interruption (§05) — "Stopped by the system", "Force-stopped", or "Clips kept after an interruption" — from the real terminated/stopReason. The user learns why it's short.
  • Kept-raw (MULTI_SEGMENT_KEPT): Basics report what was kept — "Length kept", "Clips kept", size of the kept segments — not a phantom full length. The values come from the segments that actually exist.
  • "Combined file: not created" is shown as an honest italic-muted row when no merged output exists — explaining, in the user's terms, why this plays as separate clips.
  • No recovery jargon (DiscardEligibility, TerminalAction) reaches the user; those are internal. Info translates outcome → plain language.
Unavailable fields — the presentation rule
  • Known-absent by design (fps/codec/bitrate/location/device): the row does not appear at all. Absence of a row Rova never fills is quieter and more honest than "unknown" on every recording.
  • Expected-but-missing for this recording (e.g. combined file for a kept-raw session, a side not finalized): shown as an italic muted value ("not created", "not finalized") — the row is meaningful, the gap is the information.
  • Never fabricate, never blank. No estimated durations, no "0 B" standing in for unknown, no guessed resolution.
  • If the manifest re-read itself fails, Info shows the ALREADY-LOADED basics (from Ready) and omits the manifest-only groups — it degrades, it doesn't error.
Frozen decision

Interruptions are explained, not hidden: the banner states the cause and Basics report what was actually kept. Unavailable information follows one rule — never-captured fields are omitted entirely; expected-but-missing fields show an honest muted placeholder; nothing is ever fabricated. A partial manifest degrades to the Ready-backed basics rather than failing. Info's job is to make an imperfect recording legible, calmly.

09

Motion — reuse the sheet vocabulary, add nothing

Info introduces no new motion language; it is the same pinned sheet the gesture/sharing specs already move.
Motion (inherited only)
  • Sheet enter/exit on the inherited container/exit timings (--t-container / --t-exit, --ease-std) and the same swipe-down-dismiss the gesture spec defines — identical to the sharing side sheet.
  • No content animation. Rows appear with the sheet; nothing staggers, counts up, or transitions on its own.
  • Chrome interaction: opening Info pauses chrome auto-hide (it's an intentional interaction), consistent with the action framework.
  • No new motion token. A row that seems to need motion needs a player-core change first.
Why so little

Info is a read. Motion here would be decoration on static text — exactly the noise the philosophy rejects. The one motion that matters (the sheet arriving) is already defined and shared with sharing; Info simply reuses it so the two sheets feel like one idiom.

10

Reduced motion — animation disappears, information does not

The inherited invariant, extended to Info.
  • Under rememberReduceMotion(), the sheet appears without its slide — it is simply present. Dismiss is instant.
  • Every field, group, banner, and the dismiss control remain in the same place, reachable by the same tap and a11y path.
  • No part of Info is discoverable only through motion — there was none to begin with, so reduced motion changes nothing about legibility.
  • Same rule player-gestures / player-actions / player-sharing apply, extended here: animation is the only thing that changes; behavior and content do not.
11

Frozen boundary — what this spec owns, what stays elsewhere

Info owns what information is shown after Info is pressed. Nothing leaks into or out of the neighboring specs.
player-info.html OWNS
  • The contents of the Info sheet after Info is pressed: which fields, their grouping, and their order.
  • The field inventory and each field's source + availability class (§03) — the "render from Ready + one cheap manifest re-read" contract.
  • The provenance mapping (Terminated × StopReason → plain language) and the recovery / kept-raw explanation.
  • The DualShot Angles representation (describe both, reconstruct nothing).
  • The technical-metadata policy (requested quality shown; fps/codec/bitrate rejected; actual resolution deferred), the unavailable-field rule, and the Info a11y + motion contract.
NOT owned here (do not design)
  • The Info affordance itself — placement, target size, presence rule — is player-actions.html. This spec only fills it.
  • player-sharing.html — the share flow (Info never shares; it describes).
  • player-editing.html — any field that would require a re-render or new capture.
  • player-dualshot.html — angle swap / dual playback controls (Info only lists the two angles).
  • Actual-resolution decode, device model, location — future capability, each its own decision/ADR, not smuggled in here.
  • MediaSession / cloud / accounts — no backend exists; out of scope everywhere.
Boundary rule

This spec may consume the Info action player-actions defined, the identity PlayerUriResolver mints, and the manifest SessionStore loads — but it may not redefine any of them. The information contract — provenance-first, render from Ready + one cheap manifest re-read, requested-quality-only, reject-don't-fabricate; a vaulted session opens Info like any resolved artifact (read-only, FLAG_SECURE-protected) and whether the Info action is present is player-actions' rule, not this spec's — is frozen here; a change routes back through this HTML and is re-approved first, exactly like every other change in the HTML-first workflow. Info remains PROPOSED until the owner signs off the Info action in player-actions; both gate together.
