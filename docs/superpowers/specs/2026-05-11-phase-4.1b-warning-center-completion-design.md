# Phase 4.1b — WarningCenter completion: deferred rows + hard-block Start-gate

> **Status:** design approved by owner 2026-05-11. Next: implementation plan via `superpowers:writing-plans`.

**Goal:** Finish WarningCenter *coverage* by wiring the 3 rows 4.1 deferred (#1 camera-permission denied, #3 storage-insufficient-to-start, #12 microphone denied / video-only), add hard-block *enforcement* (the Record Start button is disabled while camera-permission is denied or storage is insufficient), and fold the replan's 16-row precedence table into `docs/WarningCenterContract.md` as a Phase-1.D revision.

**Slice boundary:** 4.1b is the slice in this spec. The snooze/dismiss model + thermal/storage hysteresis (WarningCenterContract §5) is a separate later slice **4.1c** with its own brainstorm — NOT in this spec.

**Architecture:** Three new leaf signals on `RovaApp` (two trivial permission signals mirroring `NotificationPermissionSignal` minus the SDK gate; one `StorageSignal` whose computation mirrors the recording service's existing start-time preflight). `WarningPrecedence.resolve()` grows from 6 to 9 parameters; the precedence walk gains rows #1/#3/#12 in their locked-table slots; `WarningCenterViewModel`/`aggregate()`/`WarningCenter`'s VM factory thread the 3 new flows. `RecordScreen` reads the camera-permission and storage signals directly (not through the WarningCenter VM) and ORs them into the Start button's `enabled`. No `service/**` changes. No new composable. No `Modifier.blur`.

**Tech stack:** Kotlin, Jetpack Compose, kotlinx-coroutines `StateFlow`, AndroidX Lifecycle, Accompanist Permissions, JUnit4 unit tests (no Robolectric, no Turbine). minSdk 24.

## 1. Background and prior decisions

Phase 4.1 (merged to master @ `659f6b6`, PR #12) shipped `WarningCenter`: `WarningId` (16 rows, declaration order == precedence == the replan's locked "Banner precedence" table), `WarningTier{HARD_BLOCK,CRITICAL,ADVISORY}` (banner chrome only — error-container vs secondary-container — *not* the priority axis), pure `WarningPrecedence.resolve(...)`, thin `WarningCenterViewModel` (`combine` + `stateIn`, logic in `companion fun aggregate`), `WarningCenter()` composable + 16-arm `bannerContent`, and a new `BatteryOptimizationSignal`. **13 of 16 rows are wired**; rows #1/#3/#12 ship as **inert stencils** — their `WarningId` values and `bannerContent` arms exist, but `WarningPrecedence.resolve()` has no parameter for them, so it never returns them. 4.1b makes those 3 rows live — purely additive.

Decisions carried from 4.1 (owner-signed, recorded in `docs/superpowers/specs/2026-05-11-phase-4.1-warning-center-design.md` §4) that constrain 4.1b:

- **The replan's flat 16-row precedence table is authoritative**, not the Contract's category-urgency ordering (§3/§6.1 of `WarningCenterContract.md`). 4.1b folds the table into the Contract (§6 of this spec).
- **Exact-alarm (#2) is a flat, non-gating banner.** Recording still runs without exact alarms — it falls back to inexact alarms and drifts. So #2 does *not* disable Start. 4.1b does not touch `RecordScreen`'s `onStart` for exact-alarm.
- **No `Modifier.blur`, no full-screen overlay** (WarningCenterContract NO-GO #3). 4.1b's Start-gate is "inline banner + disabled button", not a `FullCardOverlay`.
- **`WarningCenterViewModel` output is `StateFlow<WarningId?>`** (not `StateFlow<WarningCenterUiState>`).
- **Resolution errors log and degrade to `null`** (NO-GO #6) — `aggregate()` keeps the `runCatching{}.getOrElse{ Log.w; null }`.

New decisions made during the 4.1b brainstorm (owner-signed 2026-05-11):

- **Slicing:** 4.1b = rows #1/#3/#12 + hard-block Start-gate + the doc fold. 4.1c (later) = snooze/dismiss + hysteresis.
- **Start-gate set:** `CAMERA_PERMISSION_DENIED` (#1) and `STORAGE_INSUFFICIENT` (#3) disable the Record Start button. `EXACT_ALARM_DENIED` (#2) does not (carried from 4.1).
- **Start-gate UX:** the existing inline WarningCenter banner stays where it is; the Start button is rendered disabled while a gating warning is active. No new composable, no overlay.
- **Storage "insufficient to start" definition:** estimate vs free space — `StorageEstimator.estimatePeakBytes(...)` from current `RovaSettings` vs `StatFs(...).availableBytes`, mirroring the service's `onStartCommand` preflight (same 10-loop horizon for ∞ loops, same default tier). Not a fixed floor.
- **Start-gate home:** computed inline in `RecordScreen` from the two signals read directly — not derived in `RecordViewModel`, and **not** read off `activeWarning.gatesStart` (see §5 for why that would be a bug).

## 2. New leaf signals (`app/src/main/java/com/aritr/rova/ui/signals/`)

All three follow the established leaf-signal contract: a `() -> X` constructor seam (so unit tests need no `Context`), a `forContext(context)` companion factory for production, a `MutableStateFlow` private backing field exposed read-only, a `refresh()` that re-reads the seam and republishes (idempotent — `MutableStateFlow` dedupes equal values). All three are `val …Signal by lazy { …forContext(this) }` props on `RovaApp` (so cold-start paths pay nothing).

### 2.1 `CameraPermissionSignal`

`NotificationPermissionSignal`'s shape minus the `sdkInt` parameter — `Manifest.permission.CAMERA` is API 1, no version gate needed (this is the `BatteryOptimizationSignal` shape).

```kotlin
package com.aritr.rova.ui.signals

import android.Manifest
import android.content.Context
import android.content.pm.PackageManager
import androidx.core.content.ContextCompat
import kotlinx.coroutines.flow.MutableStateFlow
import kotlinx.coroutines.flow.StateFlow
import kotlinx.coroutines.flow.asStateFlow

/**
 * Phase 4.1b — leaf signal: is CAMERA permission currently granted?
 * `true` = granted = good (the WarningCenter raises no camera-permission
 * banner; the Record screen leaves Start enabled). CAMERA is API 1 — no
 * SDK gate. Refresh contract: poll from the host Activity's ON_RESUME and
 * whenever the in-app permission state changes (an in-app grant does not
 * pause the Activity). Ctor seam keeps the unit test off a real Context.
 */
class CameraPermissionSignal(
    private val isGranted: () -> Boolean
) {
    private val _state: MutableStateFlow<Boolean> = MutableStateFlow(isGranted())
    val state: StateFlow<Boolean> = _state.asStateFlow()

    /** Re-read the grant and publish if changed. Call from ON_RESUME / on permission-state change. */
    fun refresh() { _state.value = isGranted() }

    companion object {
        fun forContext(context: Context): CameraPermissionSignal {
            val app = context.applicationContext
            return CameraPermissionSignal(isGranted = {
                ContextCompat.checkSelfPermission(app, Manifest.permission.CAMERA) ==
                    PackageManager.PERMISSION_GRANTED
            })
        }
    }
}
```

`RovaApp`: `val cameraPermissionSignal: CameraPermissionSignal by lazy { CameraPermissionSignal.forContext(this) }` (placed near the other signal props; KDoc cites NEW_UI_BACKEND_REPLAN row 1).

### 2.2 `MicrophonePermissionSignal`

Identical shape, `Manifest.permission.RECORD_AUDIO`. KDoc notes the relationship to `AudioMode.VIDEO_ONLY` (defined in `data/SessionManifest.kt`): the recording service sets `currentAudioMode = VIDEO_ONLY` when RECORD_AUDIO is denied at start; this signal surfaces the same fact pre-flight (idle: "clips will be video-only") and mid-session (confirms it). ADVISORY tier — never gates Start. `RovaApp.microphonePermissionSignal`.

### 2.3 `StorageSignal`

Different shape — it computes from current settings + free disk, not a single boolean read. The computation **mirrors the recording service's start-time preflight** (`RovaRecordingService.onStartCommand`, the check around `StorageEstimator.estimatePeakBytes(...)` ~line 626) so the banner predicts exactly what the service would reject.

```kotlin
package com.aritr.rova.ui.signals

import android.content.Context
import kotlinx.coroutines.flow.MutableStateFlow
import kotlinx.coroutines.flow.StateFlow
import kotlinx.coroutines.flow.asStateFlow

/**
 * Phase 4.1b — leaf signal: is there too little free space to START a
 * session with the current settings? Mirrors the recording service's
 * onStartCommand pre-flight: estimate peak bytes for (duration, loopCount,
 * resolution) via StorageEstimator (10-loop horizon for indefinite loops,
 * service default export tier), compare to StatFs available bytes;
 * insufficient == estimate > available. If the videos root is unavailable,
 * report `false` (cannot estimate — never false-block). Ctor seam keeps
 * the test off real disk / settings. Refresh contract: poll from the host
 * Activity's ON_RESUME and after any Record-screen settings change (so the
 * banner tracks duration/resolution/loop edits live).
 */
class StorageSignal(
    private val computeInsufficient: () -> Boolean
) {
    private val _insufficientToStart: MutableStateFlow<Boolean> = MutableStateFlow(computeInsufficient())
    val insufficientToStart: StateFlow<Boolean> = _insufficientToStart.asStateFlow()

    /** Re-run the estimate and publish if changed. Call from ON_RESUME / after a settings change. */
    fun refresh() { _insufficientToStart.value = computeInsufficient() }

    companion object {
        /**
         * Production wiring. The closure reads RovaSettings + the app's
         * videos root + StatFs at each refresh(). It captures the
         * application context only.
         */
        fun forContext(context: Context): StorageSignal {
            // Implementation note for the plan: read RovaSettings(app) for
            // durationSeconds / loopCount / resolution; resolve the videos
            // root the same way the service does; use StatFs(root).availableBytes;
            // call StorageEstimator.estimatePeakBytes(...) with the same
            // arguments and tier the service preflight uses, substituting
            // StorageEstimator.INDEFINITE_LOOP_PREFLIGHT_HORIZON for loopCount
            // when the configured loop count means "indefinite". Return
            // estimate > available, or false if the root is null/unavailable.

}
```

> The `forContext` body is left for the implementation plan to fill from the actual `StorageEstimator.kt` API and the service preflight — the spec fixes the *behavior* (mirror the preflight; null-root → false), not the exact method signatures. The `StorageSignal` *class* (seam, `insufficientToStart`, `refresh()`) is fixed as written above; the test drives the seam, so the unfilled `forContext` does not block the tests.

## 3. `WarningPrecedence.resolve()` and `aggregate()`

### 3.1 `resolve()` — 6 → 9 parameters

Three new boolean parameters; parameter order follows the precedence walk for readability.

```kotlin
internal object WarningPrecedence {
    fun resolve(
        cameraPermissionGranted: Boolean,      // #1  NEW
        exactAlarmGranted: Boolean,            // #2
        storageInsufficient: Boolean,          // #3  NEW
        thermal: ThermalStatus,                // #4 / #5 / #6 / #11 / #15
        power: PowerState,                     // #7 / #10 / #14
        camera: CameraSignalState,             // #8 / #9
        microphonePermissionGranted: Boolean,  // #12 NEW
        notificationsGranted: Boolean,         // #16
        batteryOptimizationExempt: Boolean     // #13
    ): WarningId? {
        if (!cameraPermissionGranted) return WarningId.CAMERA_PERMISSION_DENIED            // #1
        if (!exactAlarmGranted) return WarningId.EXACT_ALARM_DENIED                        // #2
        if (storageInsufficient) return WarningId.STORAGE_INSUFFICIENT                     // #3
        when (thermal) {                                                                  // #4 / #5 / #6
            ThermalStatus.SHUTDOWN -> return WarningId.THERMAL_SHUTDOWN
            ThermalStatus.EMERGENCY -> return WarningId.THERMAL_EMERGENCY
            ThermalStatus.CRITICAL -> return WarningId.THERMAL_CRITICAL
            else -> Unit
        }
        val pct = power.percent
        if (pct != null && pct < 5 && !power.charging) return WarningId.BATTERY_CRITICAL  // #7
        when (camera) {                                                                   // #8 / #9
            CameraSignalState.IN_USE -> return WarningId.CAMERA_IN_USE
            CameraSignalState.DISABLED -> return WarningId.CAMERA_DISABLED
            else -> Unit
        }
        if (pct != null && pct < 15 && !power.charging) return WarningId.BATTERY_LOW       // #10
        if (thermal == ThermalStatus.SEVERE) return WarningId.THERMAL_SEVERE               // #11
        if (!microphonePermissionGranted) return WarningId.MICROPHONE_DENIED              // #12 NEW
        if (!batteryOptimizationExempt) return WarningId.BATTERY_OPTIMIZATION_ON           // #13
        if (power.powerSaveMode) return WarningId.POWER_SAVE_MODE                          // #14
        if (thermal == ThermalStatus.MODERATE) return WarningId.THERMAL_MODERATE          // #15
        if (!notificationsGranted) return WarningId.NOTIFICATIONS_DENIED                  // #16
        return null
    }
}
```

The pure-function posture (no Android, no flows, no coroutines — JVM-testable directly) is unchanged. After 4.1b all 16 `WarningId` values are reachable from `resolve()`.

### 3.2 `aggregate()` — 9 source flows

`WarningCenterViewModel`'s constructor gains 3 `StateFlow<Boolean>` parameters; `aggregate()` gains the matching 3 `Flow<Boolean>` parameters. There are now 9 upstream flows: six booleans (`cameraPermissionGranted`, `exactAlarmGranted`, `storageInsufficient`, `microphonePermissionGranted`, `notificationsGranted`, `batteryOptimizationExempt`) plus `thermal`, `power`, `camera`. kotlinx-coroutines provides typed `combine` overloads only up to **5** flows, so:

1. `combine` five of the six booleans → a private 5-field holder (a `data class` local to the companion, or `combine(vararg)` + index access — the plan picks; a typed `data class` is preferred for clarity).
2. `combine(holder, sixthBoolean, thermal, power, camera) { … }` — a 5-arg typed `combine` — unpacks the holder and calls `WarningPrecedence.resolve(...)` inside the existing `runCatching { … }.getOrElse { e -> Log.w("WarningCenter", "warning resolution failed", e); null }`.

`stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000L), null)` is unchanged. `WarningCenter()`'s `viewModelFactory { initializer { … } }` wires the 3 new flows: `app.cameraPermissionSignal.state`, `app.storageSignal.insufficientToStart`, `app.microphonePermissionSignal.state`. `bannerContent` already has arms for `CAMERA_PERMISSION_DENIED` / `STORAGE_INSUFFICIENT` / `MICROPHONE_DENIED` (shipped as stencils in 4.1) — keep the copy and the `APP_DETAILS_SETTINGS` action target as stenciled; no `WarningCenter.kt` changes other than the factory wiring. `WarningCenter()`'s public signature stays `WarningCenter(modifier: Modifier = Modifier)` — it remains self-contained (the Start-gate does not go through it; see §4).

## 4. RecordScreen — the hard-block Start-gate

`RecordScreen.kt` already: holds `val rovaApp = remember { context.applicationContext as? RovaApp }` (nullable); mounts `WarningCenter()` in the top `Column`; has a `LifecycleEventObserver` whose `ON_RESUME` arm (added in 4.1) refreshes `notificationPermissionSignal` / `thermalStatusSignal` / `powerSignal` / `batteryOptimizationSignal`; holds `permissionsState = rememberMultiplePermissionsState(CAMERA, RECORD_AUDIO, [POST_NOTIFICATIONS])`; dispatches Start via `RecordViewModel.startPeriodicRecording()` and renders the Start button inside `RecordIdleDock`.

4.1b changes, all inside the existing `rovaApp != null` paths:

1. **Collect the two gating signals** — `val cameraGranted by app.cameraPermissionSignal.state.collectAsStateWithLifecycle()` and `val storageInsufficient by app.storageSignal.insufficientToStart.collectAsStateWithLifecycle()` (when `rovaApp == null`, treat both as the non-blocking value).
2. **Compute the gate** — `val startBlocked = !cameraGranted || storageInsufficient`.
3. **Disable Start** — thread `startBlocked` (or `startEnabled = !startBlocked && <existing enable conditions>`) into `RecordIdleDock` and onto the Start button's `enabled`. The button renders disabled/greyed; the inline `WarningCenter` banner (already mounted) shows row #1 or #3 with its `APP_DETAILS_SETTINGS` action as the fix path. The plan finds the exact parameter to thread through `RecordIdleDock`.
4. **`ON_RESUME` refreshes** — add `it.cameraPermissionSignal.refresh()`, `it.microphonePermissionSignal.refresh()`, `it.storageSignal.refresh()` to the existing `ON_RESUME` arm (joining the 4 from 4.1).
5. **Permission-state-change refresh** — a `LaunchedEffect` keyed on the permission statuses (e.g. `permissionsState.permissions.map { it.status }`) calls `rovaApp?.let { it.cameraPermissionSignal.refresh(); it.microphonePermissionSignal.refresh() }`, so an in-app grant updates the signal without an Activity pause.
6. **Settings-change refresh** — after any Record-screen settings commit (preset chip, duration/interval/resolution/loop change), call `rovaApp?.storageSignal?.refresh()`. The plan locates the commit site(s); if there are several, factor one local helper (`fun onSettingsChanged() { … ; rovaApp?.storageSignal?.refresh() }`).

Untouched: `onStart` (no exact-alarm gate), `RecoveryEchoBanner`, `RecordStatusStrip`, `RecordStatusStrip`'s callers, the `WarningCenter()` mount point, the `onStart`/`onStop`/`onStart`-camera-preview lifecycle arms.

## 5. `WarningId` — optional `gatesStart` doc property

Add a second constructor parameter `val gatesStart: Boolean = false`; `CAMERA_PERMISSION_DENIED` and `STORAGE_INSUFFICIENT` get `gatesStart = true`; everything else defaults to `false` (note: `EXACT_ALARM_DENIED` is `tier = HARD_BLOCK` but `gatesStart = false` — `tier` is chrome, `gatesStart` is behavior; they're orthogonal). `WarningIdOrderTest` pins the `gatesStart` value of every row alongside its `tier`.

**This property is documentation/discoverability only.** `RecordScreen` must NOT compute the gate from `activeWarning?.gatesStart` — that would be wrong when a higher-precedence non-gating warning is active simultaneously with storage-insufficient. Concretely: if both `EXACT_ALARM_DENIED` (#2) and `STORAGE_INSUFFICIENT` (#3) hold, the banner shows #2 (higher precedence), so `activeWarning == EXACT_ALARM_DENIED`, `gatesStart == false` — but storage *is* insufficient and Start *should* be disabled. Reading the two underlying signals directly (§4) is correct in every combination. (#1 camera-permission, ordinal 0, is the highest-precedence warning of all, so when it holds the banner does show it — but the same robustness argument applies, and consistency wins.)

If the owner prefers to skip this property entirely, drop it — it changes nothing functional. Default: include it.
