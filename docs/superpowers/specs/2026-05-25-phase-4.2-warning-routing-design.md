# Phase 4.2 — Route warnings to History + Settings screens (design spec)

**Date:** 2026-05-25
**Status:** Draft (pending owner review)
**Prior art:** [[ADR-0007]] record warning sheets · [[ADR-0013]] warning re-skin v3 chrome canon · [[ADR-0015]] storage-full echo · [[ADR-0016]] thermal autostop · [[ADR-0017]] recovery merge architecture
**Mockup:** `mockups/new_uiux/07-warnings.html` (gitignored)
**Master baseline:** tip `1485cf5`, 1176 tests, 51W lint, 17 ADRs

## 1. Overview

Phase 4 shipped 20 `WarningId` entries that render on the Record screen only. Phase 4.2 widens the surface so that screens with their own context (History, Settings) also render the subset of warnings relevant to them. Reuses the shipped v3 chrome end-to-end; no new sheet bodies, no new ADR (this is surface expansion, not new architecture).

## 2. Scope

**In scope:**
- History screen — top warning strip above the existing recovery cards / clip grid
- Settings screen — new "Permissions & status" section at top, rendering warning chips for permission and reliability state

**Out of scope (parked for future slices):**
- Onboarding flow (3 walkthrough slides + 4 permission screens — mockup `08-onboarding.html`). Separate UX flow with `onboardingCompleted` persistence + nav-graph changes. Future Phase 4.4.
- Library row enumeration for `MULTI_SEGMENT_KEPT` sessions (Phase 4.3 carry-over).
- Inline-after-row warning chrome (was considered but only 1 of 7 Settings IDs has a matching row; replaced with the "Permissions & status" section approach).
- Cross-surface snooze coordination (today's `SNOOZE_FOREVER` persistence still wins globally — Phase 4.2 does not add new persistence semantics).

## 3. Architectural decisions

### 3.1 Routing target enum

Add new enum `WarningScreen { Record, History, Settings }` in `app/src/main/java/com/aritr/rova/ui/warnings/WarningScreen.kt`.

This is **not** the same as the existing `WarningSurface { HardBlockSheet | SoftSheet | AdvisorySheet | TopBanner }` enum (which describes visual chrome, not screen routing). Naming chosen to avoid overload.

### 3.2 Per-screen allowlists

Two constant `Set<WarningId>` exposed from `WarningScreen.kt`:

```kotlin
internal val HISTORY_WARNINGS: Set<WarningId> = setOf(
    WarningId.STORAGE_INSUFFICIENT,        // #3 — affects ability to record + export
    WarningId.STORAGE_FULL_AUTOSTOPPED,    // #12 — echo from past auto-stop, library context
    WarningId.THERMAL_AUTOSTOPPED,         // #13 — same echo pattern
    WarningId.CANT_MERGE,                  // #14 — recovery merge pre-flight failure
    WarningId.NOTIFICATIONS_DENIED,        // #20 — export visibility relevance
)

internal val SETTINGS_WARNINGS: Set<WarningId> = setOf(
    WarningId.CAMERA_PERMISSION_DENIED,    // #1
    WarningId.EXACT_ALARM_DENIED,          // #2
    WarningId.STORAGE_INSUFFICIENT,        // #3
    WarningId.MICROPHONE_DENIED,           // #16
    WarningId.BATTERY_OPTIMIZATION_ON,     // #17
    WarningId.POWER_SAVE_MODE,             // #18
    WarningId.NOTIFICATIONS_DENIED,        // #20
)
```

Record allowlist is conceptually all 20 IDs (no allowlist applied — Record always uses the precedence-resolved `activeWarning`).

### 3.3 VM extension — multi-active per surface

`WarningCenterViewModel` today resolves a single `activeWarning: StateFlow<WarningId?>` via `WarningPrecedence.resolve(...)`. This stays unchanged (hard invariant: Record screen reads this).

Add a new public method:

```kotlin
fun activeWarningsFor(screen: WarningScreen): StateFlow<List<WarningId>>
```

Returns the ordinal-sorted list of currently active warnings filtered to the screen's allowlist. The list reflects "which of my allowed WarningIds are currently active" — not a single precedence pick. Empty list when none are active.

Implementation: derives from the same `aggregate(...)` combine pipeline the existing `activeWarning` consumes. Today that pipeline collapses to a single `WarningId?` via `WarningPrecedence.resolve(...)`. Phase 4.2 either (a) adds a second derivation branch that emits the full `Set<WarningId>` of currently-active warnings before precedence collapse, or (b) extends `WarningPrecedence` with a `resolveAll(...)` that returns all matched IDs ordinal-sorted. Implementer picks per plan task — both are JVM-testable. The new flow runs in the existing `viewModelScope` with `SharingStarted.Eagerly` matching `activeWarning`.

Snooze handling: respect `_snoozedForever` the same way `activeWarning` does — snoozed warnings are excluded from the per-surface list too.

### 3.4 History strip — per-session in-memory dismiss

History strip cards have an X button. Dismissing removes the WarningId from the strip for this session only — no persistence. Reappears on next app launch (or when the user navigates away and back, if the VM is destroyed by Compose).

Implementation: a separate `MutableStateFlow<Set<WarningId>>` on `WarningCenterViewModel` (default empty) — `historyStripDismissedThisSession`. `activeWarningsFor(History)` excludes IDs in this set.

`SNOOZE_FOREVER` warnings stay snoozed across History too — the per-session set is additive, not a replacement.

### 3.5 Settings chips — not dismissable

Settings chips have no X button. The only way to make a chip disappear is to fix the underlying signal (toggle the setting, grant the permission, free storage).

### 3.6 CTAs reuse existing sheet content

Tapping a History strip card or Settings chip opens the existing `WarningSheetV3` with the existing `warningSheetContent(id)` body. No new sheet bodies. The same primary/secondary/tertiary CTAs that fire on Record screen fire on these surfaces.

`WarningSheetV3` is already a standalone composable per Phase 4.3's exploration. The History strip / Settings chip composables call into it via a small launcher (a `rememberSaveable<Boolean>` sheet-open state per card or chip, keyed by `WarningId`).

## 5. UI surface details

### 5.1 History strip

Mount point: above `RecoveryCardList` in both `HistoryScreen.kt` branches (the empty-library `Column` branch and the non-empty `LazyColumn` branch — in the latter, an `item(key = "warning-strip")` before the existing `item(key = "recovery-cards")`).

Visual: vertical stack of advisory-tier cards. Each card uses the existing v3 chrome (severity tag · short message · primary CTA · X dismiss button). Tap on body opens the v3 sheet via the same routing the Record screen uses. Tap on X removes the card from this session's strip.

Empty list = the composable returns `Unit` (no spacer, no empty header). The composable also returns early if `WarningCenterViewModel` hasn't been wired (defensive null-check).

### 5.2 Settings "Permissions & status" section

Mount point: above the existing "Recording behavior" section in `SettingsScreen.kt`.

Section header: `"PERMISSIONS & STATUS"` matching the existing section-header typography (small uppercase muted label).

Visual: vertical stack of chips, one per active WarningId in `SETTINGS_WARNINGS`. Same severity-colored left border + chip layout as the History strip cards, no X button. Tap opens the v3 sheet.

Empty list = the entire section (header + content) is hidden. Avoids showing an empty "Permissions & status" section when everything is granted.

### 5.3 Sheet content reuse

Both surfaces dispatch sheet opens via the existing `warningSheetContent(id)` lookup. No new arms in `WarningSheetContent.kt`. No changes to `WarningSheetV3.kt`.

CTAs that target system Intents (`STORAGE_SETTINGS`, `APP_DETAILS_SETTINGS`, `EXACT_ALARM_SETTINGS`, etc.) launch via the same `launchActionTarget` already in `WarningCenter.kt`. The recovery-specific `KEEP_SEGMENTS_ONLY` / `DISCARD_RECOVERY_SESSION` targets are only relevant when `CANT_MERGE` is active, which is reachable from History — the existing dispatch path passes `pendingCantMergeSessionId` per Phase 4.3's wiring.

### 5.4 Composable not generalized today

The existing `WarningCenter(...)` composable takes `hudState: RecordHudState` — Record-screen-specific. Phase 4.2 does NOT generalize `WarningCenter`; instead it introduces the two new composables (`HistoryWarningStrip`, `SettingsPermissionsSection`) which call `WarningSheetV3` and `launchActionTarget` directly without going through `WarningCenter`. This keeps Record's chrome and routing untouched.

## 6. Dismissibility rules

| Surface | Dismissable? | Persistence | Mechanism |
|---|---|---|---|
| Record | Yes (per existing rules) | `SNOOZE_FOREVER` for permission-style IDs persists via `RovaSettings.snoozedForever`. Echoes persist via `dismissedAutoStopEchoIds`. | Unchanged from current ship |
| History strip | Yes (X button per card) | None — in-memory only, cleared on VM destruction | New `historyStripDismissedThisSession: MutableStateFlow<Set<WarningId>>` on VM |
| Settings chip | No | n/a | Chip disappears only when the underlying signal flips |

`SNOOZE_FOREVER` from Record still applies globally — a snoozed warning is excluded from History and Settings flows too. This is a deliberate user-promise: snoozing somewhere snoozes everywhere.

## 7. Failure modes

| Scenario | Behavior |
|---|---|
| `WarningCenterViewModel` not constructable (e.g. `videosRoot == null`) | Both new composables receive null/empty input and render nothing. No crash. |
| User dismisses on History, then signal flips off+on again | Card reappears (in-memory set is keyed by ID; new signal activation is a fresh emission but the dismissed-set is by ID — needs to clear on signal-down-edge). See §7.1 below. |
| `STORAGE_INSUFFICIENT` active simultaneously with `CANT_MERGE` on History | Both render, ordinal-sorted: STORAGE_INSUFFICIENT (#3) above CANT_MERGE (#14) |
| User snoozes a warning on Record, then opens History | Warning is absent from History strip too (snooze applies globally) |
| `STORAGE_INSUFFICIENT` appears on both History and Settings simultaneously | Renders on both. User dismissing on History does NOT affect Settings (Settings chips are not dismissable; History dismiss is local) |

### 7.1 Signal down-edge clears History dismissal

When a WarningId transitions from active to inactive, its History per-session dismissal should clear so the next activation re-shows it. Implementation: in the VM combine block, on each signal-down-edge for an ID in `HISTORY_WARNINGS`, remove that ID from `historyStripDismissedThisSession`.

This is the only stateful interaction beyond the trivial filter — worth a dedicated test (`historyStripDismissalClearsOnSignalDownEdge`).

## 8. Test policy

Pure JVM tests under `app/src/test/java/`. JUnit 4 + `kotlinx.coroutines.runBlocking` (project convention — no `kotlinx-coroutines-test`). `org.junit.Assert.*` assertions.

Test coverage targets (~10 new test cases):

- `WarningScreenAllowlistTest` — allowlist contents pinned, every entry exists in `WarningId`, sets are not empty
- `WarningCenterViewModelTest` extension:
  - `activeWarningsFor(History) returns ordinal-sorted list of active history-allowlist IDs`
  - `activeWarningsFor(Settings) returns ordinal-sorted list of active settings-allowlist IDs`
  - `snoozed warning excluded from per-surface list`
  - `dismissOnHistoryStrip(id) removes from history list but not settings`
  - `signal down-edge clears history dismissal`
  - `STORAGE_INSUFFICIENT active surfaces on both History and Settings simultaneously`
- `WarningCenterAggregateTest` extension:
  - `multi-active aggregation — 3 simultaneous warnings render correctly in History list`

No Compose UI tests. The composable layer is a thin renderer of the VM list.

## 9. Out-of-scope

- Onboarding flow (Phase 4.4 candidate)
- Library row enumeration for `MULTI_SEGMENT_KEPT` (Phase 4.3 carry-over)
- Generalizing `WarningCenter(...)` composable beyond Record screen (out-of-scope; new composables are cheaper than re-architecting the existing one)
- Adding new permission rows to Settings (the "Permissions & status" section sidesteps this need)
- Cross-surface snooze coordination changes (snooze-everywhere semantics unchanged)
- Notification badge counts / "you have N warnings" headers
- Animation / transition polish for chips appearing/disappearing
