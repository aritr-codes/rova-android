## 0. Scope (LOCKED)

### What this slice ships

Activates the P+L (Portrait + Landscape) Mode tab. Wires the Phase 6.1a `com.aritr.rova.service.dualrecord` package into `RovaRecordingService` so that when the user selects `mode = "PortraitLandscape"`, each segment produces two playable clips (9:16 portrait + 16:9 landscape) via the locked CameraEffect + EGL fan-out pipeline. Each merged session output publishes as an independent MediaStore entry; the History screen surfaces one card per side. Manifest schema migrates 4 → 5 with an additive `SegmentRecord.side` field; recovery scanner extends to group disk segments by `(index, side)`; storage gate math gains a mode parameter so the Start-gate row #3 (`STORAGE_INSUFFICIENT`) auto-extends to P+L's 2× peak.

### What this slice does NOT ship

- **Any modification to `service/dualrecord/**`.** The 6.1a foundation package is the source of truth; 6.1b consumes via the public API surface (`DualVideoRecorder`, `DualVideoRecorderConfig`, `DualOutput`, `DualRecording`, `DualRecordEvent`, `DualRecorderTypes.VideoSide` + the package-root `SegmentPathBuilder.build` already public via the 6.1a contract sample in §7). Verified at gate via `git diff master..HEAD -- app/src/main/java/com/aritr/rova/service/dualrecord/` returns empty.
- **Any modification to `WarningId`, `WarningPrecedence`, `WarningCenterViewModel`, or any file under `ui/warnings/**` and `ui/signals/(Camera|Microphone|Notification|ExactAlarm|Thermal|Power|BatteryOptimization|CameraState|StorageLowMidRec)Signal.kt`.** The 17-row precedence + 9-arg `WarningPrecedence.resolve` walk + leaf-signal consumers are byte-identical. `StorageSignal` itself IS modified (E1 cascade — see §4) but the modification is additive (new `mode` parameter; existing seam shape compatible via overload or default-arg per impl judgement; see Open Q OQ-1).
- **Any modification to the Start-gate expression at `RecordScreen.kt:107-122`** (`startBlocked = !cameraPermissionGranted || storageInsufficient`). Storage math becomes P+L-aware via E1 plumbing UPSTREAM of the gate; the gate expression itself is byte-identical.
- **Any modification to `flipCamera()` body** at `RovaRecordingService.kt:1178-1189`. Byte-for-byte preserved. The new dispatch in `setupCamera()` is transparent to `flipCamera` (it routes through `forceReconfigureCamera()` → `setupCamera()` → the new dispatcher).
- **Any change to `app/build.gradle.kts`, `gradle/libs.versions.toml`, `settings.gradle.kts`.** CameraX 1.4.2 + the 6.1a foundation are sufficient.
- **Any new dependency.**
- **Any whole-screen rotation.** App chrome stays portrait; only the preview composition's Compose-side overlay changes for P+L mode.
- **Mid-recording mode switch.** The existing `isRecording` guard in `setMode()` and `flipCamera()` stays. Switching to / from / between modes mid-recording remains a silent no-op (logged at debug level).
- **`DualRecording.pause()` / `.resume()`.** Carried from 6.1a's YAGNI defer. Single-mode doesn't have pause; P+L mirrors.
- **`flipCamera`-style symmetric `setMode` refactor for seamless rebind.** Seamless-rebind is a future slice. 6.1b preserves the existing silent-no-op semantics for mid-rec mode change.

### Components

#### 1.1 New helpers (private, inside existing files)

- **`RovaRecordingService.setupSingleCamera()`** — byte-identical extraction of today's `setupCamera()` body (the Recorder + VideoCapture + Preview build + `bindToLifecycle(this, currentCameraSelector, preview, videoCapture)` varargs bind path). Visibility: `private`. No behavioral change vs master.
- **`RovaRecordingService.setupDualCamera()`** — new private helper. Builds the `DualVideoRecorderConfig` (per the 6.1a spec §7 sample), instantiates `DualVideoRecorder`, builds a `Preview`, wraps both into a `UseCaseGroup` via `addUseCase(preview).addEffect(dualRecorder.asCameraEffect()).build()`, then calls `bindToLifecycle(this, currentCameraSelector, useCaseGroup)` (the `UseCaseGroup` overload). Holds `currentDualRecorder` reference for segment lifecycle + teardown. NOT bound: `VideoCapture` (dual path uses encoders via CameraEffect, not VideoCapture).
- **`RovaRecordingService.setupCamera()`** — becomes a thin dispatcher: `if (currentMode == "PortraitLandscape") setupDualCamera() else setupSingleCamera()`. The preamble (cameraProvider acquisition + mutex hold + `RovaLog.d` line + `isCameraActive` state update) is hoisted ABOVE the dispatch OR duplicated cleanly across both helpers — impl-time judgement (see Open Q OQ-2).

#### 1.2 New service-side state

- **`currentDualRecorder: DualVideoRecorder?`** — mirrors the single-mode `videoCapture: VideoCapture<Recorder>?` field. Lifetime: created in `setupDualCamera()`, released at service teardown via `currentDualRecorder?.release()`.
- **`currentDualRecording: DualRecording?`** — mirrors the single-mode `currentRecording: Recording?` field. Lifetime: created at segment start (`dualRecorder.start(...)`), nulled at segment stop. Stop calls `currentDualRecording?.stop()` which fires `DualRecordEvent.Finalize`.

#### 1.3 Segment lifecycle handler

`recordSegment()` today calls `prepareRecording().start(executor, listener)`. For P+L:

- Build `DualOutput(portraitFile, landscapeFile)` via `SegmentPathBuilder.build(sessionDir, seq, VideoSide.PORTRAIT)` + same for `LANDSCAPE` (the 6.1a `SegmentPathBuilder` is already public).
- Call `dualRecorder.start(dualOutput, executor, ::handleDualVideoEvent)`.
- `handleDualVideoEvent` receives `DualRecordEvent.Start | Status | Finalize`:
  - `Start`: parallels single-mode `VideoRecordEvent.Start` — no-op or log.
  - `Status`: parallels single-mode `VideoRecordEvent.Status` — currently unused for status display; reserved.
  - `Finalize(portraitFile?, landscapeFile?, error?)`:
    - If both files non-null + error null: persist 2 SegmentRecords (one per side); segment counter advances by 1.
    - If one file non-null (partial-success per the 6.1a per-side failure tolerance): persist the present side's SegmentRecord; log the missing side as a partial-failure anomaly; counter still advances by 1 (the ordinal is consumed regardless).
    - If both files null OR error set with both files null: classify as `SegmentResult.RetryableFailure` (existing retry budget applies) — see service `SegmentResult` sealed class.

`SegmentRecord` persistence sets `side = VideoSide.PORTRAIT` or `side = VideoSide.LANDSCAPE` for P+L segments; `side = null` for single-mode segments (preserves existing data shape).

#### 1.4 Segment counter semantics

`currentSegmentCount` increments by 1 per **ordinal**, not per file. P+L sessions with N ordinals have N entries in the segment counter, but 2N SegmentRecords in the manifest. The semantic is "completed clip count" (a clip is the unit the user thinks about), not "encoded file count".

#### 1.5 Service teardown

Service `onDestroy` + the existing `unbindAll()` paths gain a paired `currentDualRecorder?.release()` call (idempotent per 6.1a contract). Symmetric to `currentRecording = null` cleanup for single-mode.

#### 1.6 Preview Compose layer

`RecordScreen` reads `mode` from the VM. When `hasCapturePermissions` is true, the existing `AndroidView(factory = { _ -> previewView })` block wraps in a Box. For P+L mode, the Box additionally renders `SplitPreviewOverlay()` — a new private composable inside `RecordScreen.kt` (no separate file; <50 lines) that draws:

- A horizontal `Divider` at ~50% height (with the visual treatment from the mockup's `.cam-split-divider`: 1.dp white-alpha line, full width).
- Two zone-tag chips: top chip "Portrait · 9:16" anchored at top-center of the upper half; bottom chip "Landscape · 16:9" anchored at bottom-center of the lower half. Chips use the same glass-pill styling as the existing `RecordHudState` pills (see `RecordTopOverlay` styling) for consistency.

Mode-conditional: rendered only when `mode == "PortraitLandscape"`. The overlay is informational only — both halves of the screen show the same underlying full camera frame from the single `PreviewView`. The actual file outputs are the cropped MediaCodec encoder writes from the 6.1a pipeline; the preview's overlay is a UX cue telling the user "two outputs are being recorded."

`★ Open Q OQ-3:` chip placement during the active HUD states (Recording / Waiting / Merging). The active HUD already mounts loop-pill + status-pill at top-center. If P+L chips also mount at top-center, visual collision is possible. Default: render P+L chips on the **idle state only**; the active HUD's existing pills carry the user's attention. Alternative: render chips during active states too, anchored to the divider line (top chip ABOVE the line, bottom chip BELOW the line). Impl-time decision; safer default is idle-only.

#### 1.7 Picker activation

`SessionSettingsSheet.kt`'s private `ModeTab` enum currently has `PortraitLandscape("P + L", null)` where `null` is the disabled-stencil sentinel. 6.1b changes this single line to `PortraitLandscape("P + L", "PortraitLandscape")`. The picker's existing `tab.value == null` disabled-state branch resolves to `false`; the `enabled && !isDisabled` clickable gate evaluates `true`; the tap dispatches to `onModePick("PortraitLandscape")` → `viewModel.setMode("PortraitLandscape")` → service `setMode` → reconfigure to dual path. No other change to the picker composable; visual treatment for the active tab is identical to Portrait / Landscape.

### Data flow

#### 1.8 User picks P+L from picker

1. User opens `SessionSettingsSheet` (settings card tap).
2. Taps the "P + L" tab → `onModePick("PortraitLandscape")`.
3. `RecordViewModel.setMode("PortraitLandscape")` commits to `RovaSettings.mode` SharedPreferences + updates `mode: StateFlow<String>` + calls `service.setMode("PortraitLandscape")`.
4. Service: `setMode` body — `isRecording` guard (silent no-op if recording) → `currentMode = mode` → `forceReconfigureCamera()` → `unbindAll()` → `setupCamera()` → `setupDualCamera()`.
5. `setupDualCamera()`: builds `DualVideoRecorderConfig` (using `viewModel.duration / loopCount / resolution` per session settings? No — per 6.1a contract, these are decided at config-construction time using the same RovaSettings as today's single-mode path), instantiates `DualVideoRecorder`, builds Preview, builds `UseCaseGroup`, calls `bindToLifecycle(...)`. CameraX delivers frames to the dual pipeline; PreviewView's surface receives the EglRouter's preview-target render.
6. `RecordScreen` re-composes; `mode == "PortraitLandscape"` → `SplitPreviewOverlay` renders.
7. Storage gate recomputes via the `LaunchedEffect(duration, loopCount, resolution, mode)` key list (gained `mode` per E1 cascade); `STORAGE_INSUFFICIENT` row #3 reflects the 2× P+L peak. If insufficient, the existing Start-gate at `RecordScreen.kt:107-122` blocks Start (unchanged expression; new `mode`-aware leaf signal value).

#### 1.9 User taps Start in P+L mode

1. `onStart()` at `RecordScreen.kt:371-395` calls `RovaRecordingService.start(...)`. Unchanged signature.
2. Service `onStartCommand`: preflight gates (storage, permissions). Storage preflight now passes `currentMode` (just-set via `setMode`) to `StorageEstimator.estimatePeakBytes(..., mode)`. If insufficient → terminal with `StopReason.LOW_STORAGE`.
3. `createSession()` writes the manifest with `SessionConfig(..., mode = "PortraitLandscape")` (SCHEMA_VERSION = 5).
4. `setupCamera()` already routed to `setupDualCamera()` from step 4 of §1.8; the bind is live.
5. `startPeriodicRecording()` → `recordSegment()`: dispatches on `currentDualRecorder != null` (dual mode active):
   - Build `DualOutput(portraitFile, landscapeFile)` via `SegmentPathBuilder.build(sessionDir, currentSegmentCount + 1, side)`.
   - `currentDualRecording = currentDualRecorder!!.start(dualOutput, recordingExecutor, ::handleDualVideoEvent)`.
   - Wait `duration` seconds.
   - `currentDualRecording?.stop()` → `DualRecordEvent.Finalize` fires.
6. `handleDualVideoEvent.Finalize`:
   - Both files present + no error → persist 2 SegmentRecords:
     - `SegmentRecord(filename = "segment_NNNN_P.mp4", side = VideoSide.PORTRAIT, durationMs, sizeBytes, sha1)`.
     - `SegmentRecord(filename = "segment_NNNN_L.mp4", side = VideoSide.LANDSCAPE, durationMs, sizeBytes, sha1)`.
   - Persist via `sessionStore.appendSegment(...)` × 2.
   - Increment `currentSegmentCount` by 1.
7. Loop: next segment ordinal → repeat.

#### 1.10 User taps Stop / session ends

1. Service `stopPeriodicRecordingAndMerge()` (existing flow).
2. `performMerge()`: reads manifest, reads SegmentRecord list. If `manifest.config.mode == "PortraitLandscape"`:
   - `val portraitSegments = manifest.segments.filter { it.side == VideoSide.PORTRAIT }`
   - `val landscapeSegments = manifest.segments.filter { it.side == VideoSide.LANDSCAPE }`
   - Call `VideoMerger.mergeSegments(portraitSegments, ...)` → produces `<sessionDir>/merged_portrait.mp4` (or equivalent — exact filename per impl choice, see Open Q OQ-4).
   - Call `VideoMerger.mergeSegments(landscapeSegments, ...)` → produces `<sessionDir>/merged_landscape.mp4`.
   - Both merges run sequentially (simpler concurrency model; merger uses MediaMuxer which is single-threaded per-instance anyway).
3. ExportTier publish: each merged file is published independently to MediaStore.
   - Tier 1 (API 29+): `MediaStore.createPending(...)` × 2; `update(pendingUri, COMPLETED)` × 2. One side's failure does not impact the other (independent atomicity per D12).
   - Tier 2 (API 26-28): `.part` rename atom × 2.
   - Tier 3 (API 24-25): same as Tier 2.
   - Manifest fields `pendingUri` / `publicTargetPath` / `privateTempPath`: a per-side variant is needed — see Open Q OQ-5.
4. Manifest `terminated = COMPLETED`, `exportState = FINALIZED`.
5. History query reflects 2 cards for the P+L session.

#### 1.11 Recovery flow for an interrupted P+L session

1. Process restart / app cold-start; Phase 1.5 recovery runs.
2. `RecoveryScanner.classify(sessionId)`: reads manifest, reads `dir.listFiles()`.
3. Regex extension `^segment_(\d{4})(?:_([PL]))?\.mp4$` matches each disk file; captures `(index, side?)` where `side` is `PORTRAIT` for `_P`, `LANDSCAPE` for `_L`, or `null` for single-mode.
4. Internal `DiskSegment` data class gains `side: VideoSide?` field populated from group 2.
5. Grouping walks use `(index, side)` as the key instead of `(index)`:
   - `manifestByKey: Map<Pair<Int, VideoSide?>, SegmentRecord>` instead of `manifestByIndex: Map<Int, SegmentRecord>`.
   - `diskDuplicateKeys: List<Pair<Int, VideoSide?>>` — 2 disk files at `(N, PORTRAIT)` would be the genuine duplicate case; the natural P+L pair of `(N, PORTRAIT)` + `(N, LANDSCAPE)` is NOT a duplicate.
   - `missingKeys`: a manifest record at `(N, PORTRAIT)` with no disk file is the missing case (regardless of whether `(N, LANDSCAPE)` exists on disk).
   - `appendablePrefix` walks the next-expected ordinal as before, but the next-expected key for a P+L session is the next `(N, side)` pair for whichever side has the lower current max — see Open Q OQ-6 for the exact semantic.
6. The terminal write (`markTerminated`) is unchanged — mode-blind.
7. Phase 1.5 merger (the recovery-time merger path, separate from `performMerge`) ALSO splits by side and calls `mergeSegments` twice for P+L sessions — same logic as `performMerge`. Caller-side split orchestration; `VideoMerger.mergeSegments` stays mode-blind.

## 3. Data model & migration

### 3.1 Schema version

`SessionManifest.SCHEMA_VERSION = 5` (was 4). The bump exists to record that `SegmentRecord.side` may be present from v5 forward; v4 readers seeing `side = null` is the legacy/safe default and behaviorally unchanged.

### 3.2 SegmentRecord shape (v5)

```kotlin
data class SegmentRecord(
    val filename: String,
    val durationMs: Long,
    val sizeBytes: Long,
    val sha1: String,
    val side: VideoSide? = null     // NEW in v5; null = single-mode/legacy
)
```

`toJson` writes `side` only when non-null:

```kotlin
fun toJson(): JSONObject = JSONObject().apply {
    put("filename", filename)
    put("durationMs", durationMs)
    put("sizeBytes", sizeBytes)
    put("sha1", sha1)
    side?.let { put("side", it.name) }
}
```

`fromJson` reads with safe null default:

```kotlin
fun fromJson(json: JSONObject): SegmentRecord = SegmentRecord(
    filename = json.getString("filename"),
    durationMs = json.getLong("durationMs"),
    sizeBytes = json.getLong("sizeBytes"),
    sha1 = json.getString("sha1"),
    side = json.optString("side", "").ifEmpty { null }?.let {
        runCatching { VideoSide.valueOf(it) }.getOrNull()
    }
)
```

Migration: v4 manifests have no `side` field on any SegmentRecord → all read with `side = null`. Phase 6 sessions (mode = "Portrait" / "Landscape") write SegmentRecords with `side = null` (single file per ordinal, mode is on the parent SessionConfig). P+L sessions (Phase 6.1b) write SegmentRecords with `side = PORTRAIT` or `side = LANDSCAPE`.

### 3.3 SessionConfig.mode coercion

```kotlin
mode = json.optString("mode", "").ifEmpty { null }
    ?.takeIf { it == "Portrait" || it == "Landscape" || it == "PortraitLandscape" }
    ?: "Portrait"
```

Default for legacy / corrupt / unknown values: `"Portrait"`. Mirrors Phase 6's coercion shape. The display string `"P + L"` is NOT a valid persisted value — only the canonical `"PortraitLandscape"` round-trips.

### 3.4 RovaSettings.mode coercion

```kotlin
var mode: String
    get() = (prefs.getString("mode", "Portrait") ?: "Portrait")
        .takeIf { it == "Portrait" || it == "Landscape" || it == "PortraitLandscape" } ?: "Portrait"
    set(value) = prefs.edit { putString("mode", value) }
```

Same shape as Phase 6. Stale "P + L" values from earlier disabled-stencil iterations (which never wrote it, but defensive) coerce to "Portrait".

### 3.5 VideoSide import boundary

`VideoSide` is declared in `com.aritr.rova.service.dualrecord.DualRecorderTypes.kt` (per 6.1a spec §2). Importing it into `com.aritr.rova.data.SessionManifest.kt` crosses the data/service module boundary — but this is acceptable per 6.1a precedent (the 6.1a spec §7 explicitly anticipates 6.1b importing the enum at the data layer). No layering violation introduced; the enum is part of the dualrecord public API surface.

## 4. Recovery + storage extension

### 4.1 RecoveryScanner regex + grouping

```kotlin
val SEGMENT_REGEX = Regex("""^segment_(\d{4})(?:_([PL]))?\.mp4$""")
```

Group 1: index (4-digit). Group 2: side discriminator (`"P"`, `"L"`, or empty string when unmatched).

`DiskSegment` (private inside RecoveryScanner) gains `side`:

```kotlin
private data class DiskSegment(val file: File, val index: Int, val side: VideoSide?)
```

Where `side` is derived from group 2:

```kotlin
val side: VideoSide? = when (match.groupValues[2]) {
    "P" -> VideoSide.PORTRAIT
    "L" -> VideoSide.LANDSCAPE
    else -> null
}
```

### 4.2 Grouping key shift

All `groupBy { it.index }` calls in RecoveryScanner change to `groupBy { it.index to it.side }`. Manifest record indexing changes from `Map<Int, SegmentRecord>` to `Map<Pair<Int, VideoSide?>, SegmentRecord>`. Anomaly types stay byte-identical at the data shape (they carry index lists or filename lists per existing); but the underlying detection respects the (index, side) keying.

**Duplicate detection (`diskDuplicateKeys`)**: a P+L pair `(1, PORTRAIT)` + `(1, LANDSCAPE)` is NOT a duplicate (different keys). Two `(1, PORTRAIT)` files (impossible with canonical regex) IS a duplicate.

**Missing detection (`missingKeys`)**: a manifest record at `(1, PORTRAIT)` with no disk file is missing (regardless of whether the `(1, LANDSCAPE)` file exists on disk).

**Orphan detection**: a disk file at `(1, PORTRAIT)` with no manifest record is an orphan (regardless of whether `(1, LANDSCAPE)` is in the manifest).

**Appendable prefix (per-side)**: see Open Q OQ-6.

### 4.3 Phase 1.5 recovery merger

The current Phase 1.5 path (post-classification) merges classified-as-recoverable sessions. Today it calls `VideoMerger.mergeSegments(segments, output)` once. For P+L sessions (`manifest.config.mode == "PortraitLandscape"`):

```kotlin
val byMode = manifest.config.mode
if (byMode == "PortraitLandscape") {

    val landscape = manifest.segments.filter { it.side == VideoSide.LANDSCAPE }

    VideoMerger.mergeSegments(landscape, landscapeOutputFile)
} else {
    VideoMerger.mergeSegments(manifest.segments, singleOutputFile)

```

Caller-side split per D8. `VideoMerger.mergeSegments` signature unchanged.

### 4.4 StorageEstimator extension

```kotlin
fun estimatePeakBytes(
    durationSeconds: Long,
    loopCount: Int,
    resolution: String,
    tier: ExportTier,
    mode: String = "Portrait"     // default for back-compat; explicit at all callers
): Long {
    val loops = if (loopCount == -1) INDEFINITE_LOOP_PREFLIGHT_HORIZON else loopCount.toLong()
    val sides = if (mode == "PortraitLandscape") 2L else 1L
    val captureBytes = durationSeconds * loops * bytesPerSecondForResolution(resolution) * sides
    return captureBytes * tier.peakBudgetMultiplier
}
```

Default `mode = "Portrait"` is for back-compat (any unanticipated caller still works). All explicit callers (service preflight, StorageSignal) thread `mode` from the live source.

`accumulatedSessionBytes` is UNCHANGED — its broad glob `segment_*.mp4` already auto-includes paired-suffix files (per 6.1a spec §6 D6 note). Live disk pressure auto-doubles for P+L sessions with no code change.

## 8. Risks + open questions

### Risks

| Risk | Mitigation |
|---|---|
| Recovery scanner regression on existing single-mode sessions (group key change) | Test pin: every existing single-mode RecoveryScannerTest case re-runs green with `side = null` keying. The (index, null) keying is mathematically identical to the old (index) keying for single-mode-only sessions. |
| StorageSignal breaking change to its seam lambda | `StorageSignal` is constructed once (in `RovaApp.onCreate`); blast radius is bounded. Tests pin the new seam. |
| `setupCamera` byte-identical-extraction drift | Manual diff check at the plan/review stage: `git diff master..HEAD -- app/src/main/java/com/aritr/rova/service/RovaRecordingService.kt | grep -A50 "setupSingleCamera"` should show ONLY the function-header line as new; the body is the verbatim move from `setupCamera`. |
| Compose SplitPreviewOverlay layout breakage at extreme font scales | Inline composable uses `LocalConfiguration.screenHeightDp`-derived offsets; safer than fixed dp anchors. Plan-stage UI smoke verifies font scales 100% / 150% / 200%. |
| Export tier per-side atomicity edge case (one side succeeds + one fails) | Aligned with 6.1a `DualMuxer` tolerance. UX: History shows 1 card (the succeeded side); the failed side is logged but does not block the succeeded one. Manifest's failed-side fields stay null. |
| Manifest v4 → v5 forward-compat (older app build reads v5 manifest) | The schema migration is purely additive; v4 readers ignore unknown fields (per `optString` / `optBoolean` pattern). A v4 app reading a v5 manifest with `side` fields treats them as absent → may produce slightly wrong History cards but does not crash. Acceptable for forward-compat with previous app versions (rare in practice — app updates are monotonic). |
| `VideoSide` enum rename / future modification in dualrecord | The enum is on 6.1b's public-API contract surface; any future change requires a separate 6.1a-amendment slice + a 6.1b migration. Out of scope for this slice. |

### Open questions (resolved during impl or surface to owner)

- **OQ-1**: `StorageSignal` seam shape change vs additive overload. Default: lambda-signature change (`(Int, Int, String) -> Boolean` → `(Int, Int, String, String) -> Boolean`). Bounded blast radius. Plan-stage may revisit if a simpler additive shape emerges.
- **OQ-2**: `setupCamera()` preamble hoist vs duplication across helpers. Default: hoist the preamble (cameraProvider acquisition, mutex hold, log line, isCameraActive state update) above the dispatch; helpers do build + bind only. Plan-stage validates that the existing preamble has no single-vs-dual divergence.
- **OQ-3**: SplitPreviewOverlay during active HUD states (Recording / Waiting / Merging). Default: idle-only (per §5). Plan-stage UI smoke confirms no visual collision with active HUD pills.
- **OQ-4**: Merged-output filename convention for P+L. Default proposal: `merged_portrait.mp4` + `merged_landscape.mp4` in the session dir. Plan-stage may align with the existing single-mode merged filename pattern (currently `merged.mp4` per the existing service code — needs verification).
- **OQ-5**: Manifest fields for per-side publish state (`pendingUri`, `publicTargetPath`, `privateTempPath`). Default: per-side variants (`portraitPendingUri`, `landscapePendingUri`, …). Plan-stage may collapse into a single `dualOutputs` collection if the diff justifies.
- **OQ-6**: Appendable orphan prefix for P+L sessions. The single-mode logic walks `idxMaxManifest + 1, +2, …` claiming each contiguous valid orphan. For P+L, two natural interpretations: (a) walk per-side — find the lower of the two sides' max manifest indices and append per-side from there independently; or (b) require pairs — only append `(N, PORTRAIT)` + `(N, LANDSCAPE)` as a unit; if one side is missing, neither appends. - **OQ-8**: History card label copy ("Session 2026-05-15 — Portrait" vs "Session 2026-05-15 (Portrait)" vs custom). Spec-stage left to plan/UX judgement.
- **OQ-9**: Whether SCHEMA_VERSION bump (4 → 5) also needs a `currentExportTier()`-style helper for the per-side publish state, or if the existing per-tier code reads manifest fields directly. Plan-stage decides.
