# Rova WCAG 2.2 AA Accessibility Audit — Design Spec

- **Date:** 2026-05-29
- **Status:** Approved (brainstorming → ready for implementation plan)
- **Sub-project of:** the larger accessibility initiative; this spec covers the
  **audit of the existing app only**. Remediation, on-device tooling, and the
  `checkA11y*` gate implementation are later, separate cycles.
- **Conformance target:** **WCAG 2.2 Level AA** (Android-native interpretation:
  "screen reader" = TalkBack, "keyboard" = D-pad / Switch Access / external
  keyboard, "contrast" = token/color math).

## 1. Goal & non-goals

### Goal
Produce a complete, source-level accessibility audit of every user-facing Rova
surface, scored against WCAG 2.2 AA, delivered as (a) a findings report, (b) a
prioritized remediation backlog, and (c) a **draft ADR** establishing
"WCAG 2.2 AA by default" as a standing product requirement.

### Non-goals (YAGNI — explicitly out of scope this cycle)
- **No code remediation.** This cycle ships documents, not Kotlin changes.
- **No on-device tooling runs** (Accessibility Scanner / Espresso
  `AccessibilityChecks` / instrumented `semantics` tests). These are a *written
  recommendation* in the report and a future slice.
- **No `checkA11y*` gate implementation.** ADR-0020 stubs the future gate; it is
  not built here.
- **No new-feature accessibility.** "WCAG by default" applied to Settings
  expansion / custom modes is enforced during those cycles, not here.
- **Launcher icon, Play-store assets, marketing surfaces** — excluded.

## 2. Method

**Static, source-level, hybrid.** The deliverable is a 100% static code audit.
"Hybrid" = the report *recommends* on-device tooling as a follow-on slice; no
tooling is run during the audit.

Every finding is scored through **three lenses at once** so it carries both a
standard and a concrete fix:

| Lens | Role |
|------|------|
| **WCAG 2.2 AA** | The pass/fail Success Criterion (the "what's broken, by which rule"). |
| **android-skills:android-ux** | M3 10-category screen score (touch targets, spacing, accessibility, …) — the per-surface rubric. |
| **android-skills:compose** + **ui-ux-pro-max** | The concrete Compose remediation (`semantics{}`, `contentDescription`, `Modifier.minimumInteractiveComponentSize()`, `FocusRequester`/focus order, `liveRegion`, reduced-motion gating) + UX cross-check. |

These three skills are **mandatory inputs** to every surface audit, not optional.

## 3. Scope — audited surfaces

### Surface clusters (each gets one audit agent)
1. **RecordScreen** — idle layout + active-state HUD, record chrome, mode picker,
   camera guides, dual preview zone, recording frame layout.
2. **HistoryScreen** — list, history warning strip, `LibrarySessionConfigDialog`.
3. **SettingsScreen** — `SettingsSheet`, `SettingsPermissionsSection`.
4. **PlayerScreen** — Media3 surface + `SegmentedTimeline`.
5. **OnboardingScreen** — 3-slide flow.
6. **Recovery** — `RecoveryCard` + recovery UI states.
7. **WarningCenter surfaces** — `WarningSheetV3`, `WarningTopBannerV3`,
   warning/snooze chip, `ThermalTipsSheet`, `TerminalEcho`.
8. **Shared components** — steppers, selectors, dialogs, cards, chip rows
   (`RovaComponents`, `RovaDialogs`, `RovaCardComponents`, `LargeValueStepper`,
   `QualityOptionSelector`, `QuickSetChipRow`, `FixedContinuousSelector`).
9. **MainScreen** — bottom-nav scaffold + nav structure.
10. **Notification surfaces** — FGS foreground-service notification + M5 custom
    `RemoteViews` (`DecoratedCustomViewStyle`). **In scope** — it is the
    service's only persistent UI; distinct a11y model (RemoteViews
    `contentDescription`, not Compose `semantics`).

### Cross-cutting passes (each gets one agent, run once app-wide)
- **A — Contrast & target-size tokens:** math over `RovaTokens.kt`,
  `RecordChromeTokens.kt`, `Color.kt`, `Theme.kt`, `Type.kt`. Computes actual
  contrast ratios for every token pair in use; flags < 4.5:1 (normal text),
  < 3:1 (large text / non-text UI). Checks declared touch-target minimums
  against the 24×24 (SC 2.5.8) / 48dp (M3) bars.
- **B — Motion & animation:** `RovaAnimations.kt` and per-screen transitions vs
  SC 2.3.3 (animation from interactions) and 2.2.2 (pause/stop/hide); flags
  absence of a reduced-motion path.

## 4. Per-surface WCAG checklist (fixed; identical for every surface agent)

Maps the user's 10 evaluation areas → WCAG 2.2 AA Success Criteria:

| Area | SCs applied |
|------|-------------|
| Color contrast & readability | 1.4.3, 1.4.11 |
| Keyboard navigation & focus | 2.1.1, 2.4.3, 2.4.7, **2.4.11 (focus not obscured)** |
| Screen reader / TalkBack | 1.1.1, 4.1.2, **4.1.3 (status messages)** |
| Semantic structure & labeling | 1.3.1, 2.4.6 |
| Touch target sizing | **2.5.8 (24×24 min)**, M3 48dp |
| Motion & animation | 2.3.3, 2.2.2 |
| Form accessibility & validation | 3.3.1, 3.3.2, 3.3.3, **3.3.8 (accessible auth)** |
| Error states & recovery | 3.3.1, 3.3.3 |
| Responsive behavior | 1.4.10 (reflow), 1.4.4 (text resize), 1.3.4 (orientation) |
| Overall UX inclusivity | holistic (advisory) |

## 5. Findings schema & severity

### Finding row (the output contract every agent returns)
```
id · surface · WCAG SC + level · severity · what's wrong ·
file:line · Compose remediation · M3 / ui-ux cross-ref · effort (S/M/L)
```
Agents return **structured findings only**, not prose.

### Severity model (4 tiers)
- **Blocker** — fails a Level-A/AA SC outright and blocks a task.
- **Serious** — AA fail, but a workaround exists.
- **Moderate** — degraded but usable.
- **Advisory** — best-practice / cheap-AAA.

### Backlog ranking
`severity × reach`, where reach = how many users hit the surface. High-reach
surfaces (Record, History, Settings) outrank low-reach (Recovery, Onboarding).

## 6. Execution architecture

**Per-surface parallel fan-out + synthesis.**

1. **Fan-out (12 agents):** 10 surface agents + 2 cross-cutting agents. Each
   receives the identical §4 checklist and §5 schema as its output contract,
   plus the §2 three-lens skill set.
2. **Synthesis (main session):**
   - **Dedupe** — a shared component flagged on N screens collapses to one
     backlog item (with all call-sites listed).
   - **Rank** — apply severity × reach.
   - **Write** the three artifacts (§7).

> **Decision point for the plan:** a true parallel fan-out of ~12 agents is what
> the **Workflow** tool is built for, but it needs explicit owner opt-in. If not
> opted in, fall back to batched sequential `Agent`/Explore dispatch. The
> implementation plan must surface this as an explicit choice; it does not change
> the deliverable, only execution speed/cost.

Honors `CLAUDE.md`: exploration runs in subagents (CodeGraph `.codegraph/`
present) — never `codegraph_explore`/`codegraph_context` in the main session.

## 8. Documentation maintenance workstream

Per owner instruction ("keep updating the relevant documents as well"), the
audit cycle must also update the following when artifacts land:

- **`CHANGELOG.md`** — Unreleased entry: accessibility audit docs + ADR-0020 draft.
- **`CLAUDE.md`** — note the new `docs/accessibility/` location, the
  "WCAG 2.2 AA by default" convention, and the planned `checkA11y*` gate (so the
  invariant→check culture is documented before the gate exists). Bump the ADR
  count (19 → 20) and add the new ADR to the ADR summary line.
- **Roadmap** — add an "Accessibility" track entry referencing the backlog
  (target doc: `ROADMAP_v6.md` or `NEW_UI_BACKEND_REPLAN.md`, decided in the
  plan based on which track owns UI-quality work).
- **`docs/WarningCenterContract.md`** — only if a warning-surface a11y invariant
  emerges from surface-cluster 7 that belongs in the authoritative contract.
- **Project memory** (`memory/`) — record audit completion state + the standing
  WCAG policy as a `feedback`/`project` memory.

Doc updates ship **in the same PR** as the audit artifacts (this repo lands a
feature's docs with the feature).

## 9. Success criteria

- Every surface in §3 has at least one entry in the report (or an explicit
  "no findings" note) — coverage is provable.
- Every finding carries all schema fields (§5), incl. a concrete Compose fix and
  a `file:line`.
- Backlog is fully ranked; no unsized items.
- ADR-0020 draft exists and is internally consistent with the report.
- All §8 doc updates applied in the same PR.
- **Zero Kotlin source changes** (audit-only cycle — verifiable from the diff).
