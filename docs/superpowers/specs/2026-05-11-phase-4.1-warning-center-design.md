# Phase 4.1 — WarningCenter VM + Record-screen banner — design

> **Status:** Approved for implementation 2026-05-11 (owner brainstorm). This is the input spec for the Phase 4.1 dev dispatch.
> **Precedence source of truth:** `NEW_UI_BACKEND_REPLAN.md` §"Phase 4 — Warning center + integration" — the 16-row "Banner precedence" table, owner-signed 2026-05-11. That table is authoritative.
> **Surrounding model:** `docs/WarningCenterContract.md` (Phase 1.D) — owns other-surface routing (Settings / Onboarding / History), the formal persistence/snooze model, severity-token mapping, and the 18-state catalog. Where the replan's table and the Contract's §3/§6.1 ordering conflict, **the replan's table wins** (it is the later, owner-signed artifact); a later doc-hygiene pass folds the table into the Contract as a Phase 1.D revision — not part of 4.1.
> **Scope of this PR:** the aggregator + the Record-screen banner only. Routing to other surfaces is Phase 4.2; the recovery "Merge what was recorded" 3rd action is Phase 4.3.

---

## 1. Scope

Phase 4.1 builds:

- `ui/warnings/WarningId.kt` — the `WarningId` enum (16 values, declaration order = precedence) + `WarningTier` enum + `WarningId.tier`.
- `ui/warnings/WarningPrecedence.kt` — `internal object WarningPrecedence { fun resolve(...): WarningId? }` — **pure** (no Android, no flows), the testable core.
- `ui/warnings/WarningCenterViewModel.kt` — thin `combine(...).stateIn(...)` wrapper over the source `StateFlow`s, exposing `val activeWarning: StateFlow<WarningId?>`. Plus a `companion object fun aggregate(...)` flow operator carrying all the logic, so the test never touches `viewModelScope`.
- `ui/warnings/WarningCenter.kt` — the presentational layer: `WarningCenter(...)` host composable, `WarningBanner(id)` (one shared template parameterized by tier), `bannerContent(id) → BannerContent`, `WarningAction` + `ActionTarget` + `launchActionTarget(context, target)`. **All 16 rows get a `bannerContent` entry** even though only 13 are reachable in this PR (the 3 unwired rows are inert stencils until 4.1b flips their source on).
- `ui/signals/BatteryOptimizationSignal.kt` — new leaf signal, verbatim `NotificationPermissionSignal` shape minus the SDK gate. `RovaApp.batteryOptimizationSignal` lazy prop.
- `RecordScreen.kt` edits — mount `WarningCenter` in the existing top `Column`; **delete** the `BatteryOptimizationBanner` block + `showBatteryBanner` / `batteryBannerDismissed` plumbing (#13 is now a WarningCenter row); add the `ON_RESUME` → `refresh()` observer the Phase 3 signals were waiting for.

Phase 4.1 **wires 13 of the 16 precedence rows** (the rows whose producers exist today):

## 2. Architecture

### 2.1 `WarningId` + `WarningTier`

```kotlin
package com.aritr.rova.ui.warnings

/**
 * The 16 Record-screen warning banners, in PRECEDENCE ORDER (highest
 * first). Declaration order IS the contract — [WarningPrecedence.resolve]
 * returns the first active one, and [WarningIdOrderTest] pins the
 * ordinals so a reorder cannot slip through review.
 *
 * Order mirrors NEW_UI_BACKEND_REPLAN.md §"Phase 4" "Banner precedence"
 * table (owner-signed 2026-05-11), rows 1..16. Rows 1, 3, 12 are
 * present here but NOT yet reachable from [WarningPrecedence.resolve]
 * (their producers land in Phase 4.1b); they exist so the enum and
 * [bannerContent] are complete and 4.1b is additive.
 */
enum class WarningId(val tier: WarningTier) {
    // Hard block
    CAMERA_PERMISSION_DENIED(WarningTier.HARD_BLOCK),   // #1  — DEFERRED to 4.1b
    EXACT_ALARM_DENIED(WarningTier.HARD_BLOCK),         // #2
    STORAGE_INSUFFICIENT(WarningTier.HARD_BLOCK),       // #3  — DEFERRED to 4.1b
    // Critical
    THERMAL_SHUTDOWN(WarningTier.CRITICAL),             // #4
    THERMAL_EMERGENCY(WarningTier.CRITICAL),            // #5
    THERMAL_CRITICAL(WarningTier.CRITICAL),             // #6
    BATTERY_CRITICAL(WarningTier.CRITICAL),             // #7
    CAMERA_IN_USE(WarningTier.CRITICAL),                // #8
    CAMERA_DISABLED(WarningTier.CRITICAL),              // #9
    // Advisory
    BATTERY_LOW(WarningTier.ADVISORY),                  // #10
    THERMAL_SEVERE(WarningTier.ADVISORY),               // #11
    MICROPHONE_DENIED(WarningTier.ADVISORY),            // #12 — DEFERRED to 4.1b
    BATTERY_OPTIMIZATION_ON(WarningTier.ADVISORY),      // #13
    POWER_SAVE_MODE(WarningTier.ADVISORY),              // #14
    THERMAL_MODERATE(WarningTier.ADVISORY),             // #15
    NOTIFICATIONS_DENIED(WarningTier.ADVISORY)          // #16
}

enum class WarningTier { HARD_BLOCK, CRITICAL, ADVISORY }
```

`WarningTier` drives the banner chrome only (it is **not** the priority axis — the enum's ordinal is). Visual treatment per `UI_DESIGN_TOKENS.md` §2.10 `RovaWarnings.*`: `HARD_BLOCK` and `CRITICAL` → the red `Hard` token, prominent; `ADVISORY` → the amber `Soft` token, subtle. The dev picks exact tokens; this mapping is not load-bearing for the aggregator.

### 2.2 `WarningPrecedence.resolve(...)` — the pure core

```kotlin
package com.aritr.rova.ui.warnings

import com.aritr.rova.ui.signals.CameraSignalState
import com.aritr.rova.ui.signals.PowerState
import com.aritr.rova.ui.signals.ThermalStatus

/**
 * Pure resolver: given the current state of the wired signals, returns
 * the single highest-priority active [WarningId], or null if none is
 * active. No Android, no coroutines, no flows — JVM-testable directly.
 *
 * "Highest priority" = lowest [WarningId.ordinal] among the active set.
 * Implemented as a straight walk of the 16 rows in declaration order,
 * returning the first whose predicate holds.
 *
 * Rows 1 (camera-perm), 3 (storage-to-start), 12 (mic) have no
 * parameter here yet — their producers land in Phase 4.1b. Until then
 * this function never returns those three values.
 *
 * `null` charging / `null` percent semantics (PowerState): a low/critical
 * battery warning fires only when the percent is KNOWN (non-null) AND
 * below the threshold AND not charging. Unknown percent (emulator /
 * hardware quirk) → no battery warning, never a false alarm. Charging
 * is best-effort on API 24/25 (see PowerSignal KDoc) — that is
 * acceptable here: a missed "low battery" nag on a pre-Oreo device is
 * a far smaller harm than a spurious one.
 */
internal object WarningPrecedence {
    fun resolve(
        exactAlarmGranted: Boolean,
        thermal: ThermalStatus,
        power: PowerState,
        camera: CameraSignalState,
        notificationsGranted: Boolean,
        batteryOptimizationExempt: Boolean
    ): WarningId? {
        // #2
        if (!exactAlarmGranted) return WarningId.EXACT_ALARM_DENIED
        // #4 / #5 / #6
        when (thermal) {
            ThermalStatus.SHUTDOWN -> return WarningId.THERMAL_SHUTDOWN
            ThermalStatus.EMERGENCY -> return WarningId.THERMAL_EMERGENCY
            ThermalStatus.CRITICAL -> return WarningId.THERMAL_CRITICAL
            else -> Unit
        }
        // #7 — battery critical
        val pct = power.percent
        if (pct != null && pct < 5 && !power.charging) return WarningId.BATTERY_CRITICAL
        // #8 / #9 — camera. OTHER_ERROR / UNKNOWN raise nothing (per replan).
        when (camera) {
            CameraSignalState.IN_USE -> return WarningId.CAMERA_IN_USE
            CameraSignalState.DISABLED -> return WarningId.CAMERA_DISABLED
            else -> Unit
        }
        // #10 — battery low
        if (pct != null && pct < 15 && !power.charging) return WarningId.BATTERY_LOW
        // #11 — thermal severe
        if (thermal == ThermalStatus.SEVERE) return WarningId.THERMAL_SEVERE
        // #13 — battery optimization
        if (!batteryOptimizationExempt) return WarningId.BATTERY_OPTIMIZATION_ON
        // #14 — power-save
        if (power.powerSaveMode) return WarningId.POWER_SAVE_MODE
        // #15 — thermal moderate (LIGHT and NONE raise nothing)
        if (thermal == ThermalStatus.MODERATE) return WarningId.THERMAL_MODERATE
        // #16 — notifications
        if (!notificationsGranted) return WarningId.NOTIFICATIONS_DENIED
        return null
    }
}
```

Note: a row's predicate is "this row's condition is true" — the *precedence* (suppression of lower rows) falls out of the early-return walk, exactly the "re-evaluate from current signal state, no materialized queue" semantics decision #5 in the replan. `THERMAL_CRITICAL` "may stop" / `THERMAL_SHUTDOWN` "recording stopped" copy showing while no session is running is mildly imprecise but not wrong ("may"); per-scope gating (`SESSION_ONLY`) is a Phase 4.2 refinement (4.2 has to observe `serviceState` for #12 anyway), explicitly out of 4.1.

### 2.3 `WarningCenterViewModel`

```kotlin
package com.aritr.rova.ui.warnings

class WarningCenterViewModel(
    exactAlarm: StateFlow<Boolean>,
    thermal: StateFlow<ThermalStatus>,
    power: StateFlow<PowerState>,
    camera: StateFlow<CameraSignalState>,
    notifications: StateFlow<Boolean>,
    batteryOptExempt: StateFlow<Boolean>
) : ViewModel() {

    val activeWarning: StateFlow<WarningId?> =
        aggregate(exactAlarm, thermal, power, camera, notifications, batteryOptExempt)
            .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), null)

    companion object {
        /**
         * The aggregation, as a plain Flow operator over the 6 source
         * flows. Carries all the wiring so it can be unit-tested with
         * fake [MutableStateFlow]s and no [viewModelScope] — the same
         * posture as PlayerUriResolver / SegmentedTimelineMath /
         * WakeLockPolicy: the testable core is a pure-ish function, the
         * ViewModel is a 3-line `stateIn` shell.
         *
         * NO-GO #6 (WarningCenterContract §8): any throw inside the
         * combine logs and degrades to `null` — a failure to compute a
         * banner must not itself become a banner state.
         */
        fun aggregate(
            exactAlarm: Flow<Boolean>,
            thermal: Flow<ThermalStatus>,
            power: Flow<PowerState>,
            camera: Flow<CameraSignalState>,
            notifications: Flow<Boolean>,
            batteryOptExempt: Flow<Boolean>
        ): Flow<WarningId?> =
            combine(exactAlarm, thermal, power, camera, notifications, batteryOptExempt) {
                ea, th, pw, cam, nt, bo ->
                runCatching {
                    WarningPrecedence.resolve(ea, th, pw, cam, nt, bo)
                }.getOrElse { e -> Log.w("WarningCenter", "resolve failed", e); null }
            }
    }
}
```

A `viewModel { ... }` factory in `WarningCenter.kt` (or a `companion fun forApp(app: RovaApp)`) constructs it from `RovaApp`'s lazy signal props — mirroring how `RecordScreen` already pulls `rovaApp?.recoveryReport`. Under preview / unit test where `applicationContext` isn't a `RovaApp`, fall back to `MutableStateFlow`s of the "no warning" value → `activeWarning` stays `null`.

### 2.4 `BatteryOptimizationSignal`

New file `ui/signals/BatteryOptimizationSignal.kt` — copy of `NotificationPermissionSignal` shape, **no SDK gate** (`PowerManager.isIgnoringBatteryOptimizations` is API 23, minSdk is 24):

```kotlin
class BatteryOptimizationSignal(private val isExemptNow: () -> Boolean) {
    private val _isExempt = MutableStateFlow(isExemptNow())
    /** true = app is exempt from battery optimization (GOOD — no warning). */
    val isExempt: StateFlow<Boolean> = _isExempt.asStateFlow()
    /** Re-read; call from ON_RESUME (grant changes don't broadcast). Idempotent. */
    fun refresh() { _isExempt.value = isExemptNow() }
    companion object {
        fun forContext(context: Context): BatteryOptimizationSignal {
            val app = context.applicationContext
            return BatteryOptimizationSignal(
                isExemptNow = { BatteryOptimizationHelper.isIgnoring(app) }
            )
        }
    }
}
```

`RovaApp`: `val batteryOptimizationSignal: BatteryOptimizationSignal by lazy { BatteryOptimizationSignal.forContext(this) }`.

`BatteryOptimizationHelper` (`isIgnoring`, `buildRequestIntent`) is unchanged and keeps living where it is — both the signal and the `BATTERY_OPTIMIZATION_ON` banner action use it.

## 3. `RecordScreen` integration

- `WarningCenter()` renders in the existing top `Column` — the slot `BatteryOptimizationBanner` occupies today (between the top-bar `Row` and the `RecoveryEchoBanner` / `RecordStatusStrip`).
- **Delete** the `BatteryOptimizationBanner(...)` call, `showBatteryBanner`, `batteryBannerDismissed`. If `BatteryOptimizationBanner.kt` has no other caller after that, delete the file too; if Settings uses it, leave it (Phase 4.2 re-homes the Settings surface per Contract §6.4). `BatteryOptimizationHelper` stays.
- **Leave `RecoveryEchoBanner` untouched** — recovery cards are a separate surface (replan precedence-decision #5 note; Contract C4 / §6.3). 4.1 does not touch it.
- **Visibility = always when active.** `WarningCenter` shows whenever `activeWarning != null`, regardless of `hudState` (Idle / Recording / Waiting / Merging). No `hudState` gating. (`CameraSignalState` is already `UNKNOWN` when no session runs, so the camera rows self-gate to active sessions; the rest are signal-driven regardless of session.)
- **No dismiss / no snooze in 4.1.** The banner follows the signal — it clears when the underlying condition clears. The Contract §5 snooze model is explicitly deferred to 4.1b/4.2; Contract open-question #4 (snooze duration constant) stays open until then.
- **`ON_RESUME` refresh wiring** (4.1 owns this — the Phase 3 signals shipped "signal only", no callers): add a `LifecycleEventObserver` (or extend the existing one) in `RecordScreen` that, on `Lifecycle.Event.ON_RESUME`, calls `refresh()` on the 4 poll-based signals: `notificationPermissionSignal`, `thermalStatusSignal`, `powerSignal`, `batteryOptimizationSignal`. `exactAlarmSignal` already self-refreshes via the `RovaApp` broadcast receiver; `cameraStateSignal` is fed by the recording service. ## 6. Test plan

- **`WarningPrecedenceTest`** (pure JVM, no Robolectric):
  - One case per wired row: only that row's condition active → `resolve(...)` returns that `WarningId`.
  - Suppression pairs: e.g. `(exactAlarmGranted=false, thermal=SHUTDOWN)` → `EXACT_ALARM_DENIED` (#2 beats #4); `(thermal=SEVERE, batteryOptExempt=false)` → `THERMAL_SEVERE` (#11 beats #13); `(power.percent=3, charging=true)` → not `BATTERY_CRITICAL` (charging suppresses); `(power.percent=null)` → no battery warning.
  - Camera: `IN_USE` → `CAMERA_IN_USE`; `DISABLED` → `CAMERA_DISABLED`; `OTHER_ERROR` → `null`; `UNKNOWN` → `null`.
  - All-clear (`exactAlarmGranted=true, thermal=NONE, percent=80, !powerSave, camera=OK, notificationsGranted=true, batteryOptExempt=true`) → `null`.
  - Thermal `LIGHT` / `NONE` → `null` (no row).
- **`WarningIdOrderTest`** — pins `WarningId.values()` to the exact 16-name ordered list, and the `tier` of each. A reorder or a tier change fails this test.
- **`WarningCenterAggregateTest`** — drives `WarningCenterViewModel.aggregate(...)` with 6 fake `MutableStateFlow`s using the `runBlocking + launch(Dispatchers.Unconfined) { collect { emissions += it } } ; yield() ; ...mutate... ; yield() ; cancelAndJoin()` idiom (same as the Phase 3 signal tests). Asserts: initial `null` when all clear; flipping `exactAlarm` to `false` re-emits `EXACT_ALARM_DENIED`; flipping it back re-emits `null`; a `runCatching` smoke (can't easily force a throw — acceptable to skip, the path is one line).
- No new Robolectric. `WarningCenterViewModel` itself (the `stateIn` shell) and `WarningCenter.kt` (composables) are not unit-tested — same as other thin VMs / composables in this codebase; covered by the Phase 4 on-device smoke.

Gradle gate: `:app:lintDebug` (baseline unchanged at 55) + `:app:testDebugUnitTest` (unfiltered) green. Expect ~+25–30 tests over `aa06028`'s 663.
